## 3. `<!DOCTYPE html>` 是什么？`doctype` 是什么？

`<!DOCTYPE html>` 不是一个 HTML 元素，它是一种独立的语法结构，叫做「文档类型声明」（Document Type Declaration） => 写在最前面的指令，告诉浏览器进入哪种文档模式。

```html
<!DOCTYPE html>
<!-- 
它不在 DOM 树中，不是任何节点的子元素
它本质上是一个「指令」，不是页面内容的一部分
它的作用是告诉浏览器：「请用标准模式处理这个页面」。
说完这句话，它的使命就完成了。
-->
```

::: tip

`doctype` 就是 `<!DOCTYPE html>` 这个声明的简称。

两者指的是同一件事。开发社区中说「记得写 doctype」和「记得写 `<!DOCTYPE html>`」是同一个意思。

:::

## 4. 文档模式是什么？为什么由 `doctype` 触发？

### 4.1. 理解「文档模式」和 `doctype`

文档模式可以理解成浏览器解释页面时采用的兼容语境。它最早是 IE5.5 为了解决旧页面兼容问题引入的：同一个浏览器，需要既能尽量兼容旧网页，又能逐步支持更标准的行为，于是就有了模式切换。

这个切换通常由文档开头的 `doctype` 触发。浏览器看到不同的文档类型声明，会决定当前页面应该按哪一套规则去理解和渲染。`doctype` 用于告诉浏览器：这个页面应该按旧世界的兼容方式处理，还是按更标准的方式处理。

### 4.2. 浏览器渲染页面时的不同规则

浏览器在渲染页面时，并不是只有一套固定的规则。它内部准备了好几套规则，每套规则对同一种写法的处理方式不同。`doctype` 的作用就是告诉浏览器：用哪一套。

可以通过一段代码验证当前页面处于哪种文档模式：

```html
<script>
  console.log(document.compatMode)
</script>
```

- 返回 `CSS1Compat` -> 标准模式
- 返回 `BackCompat` -> 怪异模式

这两种模式在 DOM 尺寸计算、百分比高度解析、表格布局等方面存在差异，且不同浏览器的实现细节可能不同。你不需要记住每一条差异，但需要知道一个关键结论：同一种写法，不同的文档模式，浏览器的处理方式可能不同。

而 `doctype` 就是切换这套规则的开关：

```html
<!DOCTYPE html>
```

- `<!DOCTYPE html>` 表示标准模式，现代页面应该用这个
- 没有 `doctype`，或者写法不规范，浏览器可能进入怪异模式
- 在现代开发中，你只需要记住一件事：始终写上 `<!DOCTYPE html>`，浏览器就会进入标准模式
- 怪异模式是为了兼容几十年前的旧网页而保留的，现代开发中不需要、也不应该触发它

### 4.3. 由 `doctype` 触发的原因

这是一个实际工程选择，核心原因有两个：

- 第一，`doctype` 在文档中的位置足够早。浏览器需要在渲染页面之前就知道该用哪一套规则。`doctype` 恰好是 HTML 规范要求写在最前面的声明，比 `<html>` 还早。浏览器读到它的那一刻就可以做出判断，不需要额外的探测或等待。
- 第二，`doctype` 本身已经携带了「意图」信息。`doctype` 的原始用途是声明「这个页面遵循哪个 HTML 版本」。IE 的工程师在设计模式切换时，利用了这个已有的信号：
  - 如果一个页面写了规范的 `doctype`，说明作者知道标准的存在，应该用标准模式渲染。
  - 如果页面连 `doctype` 都没写，大概率是很早之前写的旧页面，应该用兼容模式。

## 5. 怪异模式、标准模式、准标准模式有什么区别？

文档模式可以分成三类：

| 模式 | 含义 | 典型特点 |
| --- | --- | --- |
| 怪异模式 | `quirks mode` | 更像早期浏览器的非标准行为，兼容历史页面 |
| 标准模式 | `standards mode` | 尽量按标准解释页面 |
| 准标准模式【不重要】 | `almost standards mode` | 大体接近标准模式，只保留少数兼容差异 |

触发方式上，可以先记两个最重要的结论：

- 省略 `doctype`，通常会进入怪异模式
- 使用现代 HTML5 的 `<!DOCTYPE html>`，通常会进入标准模式

准标准模式主要是历史过渡产物，常见于某些旧的 HTML 4.01 / XHTML Transitional、Frameset 文档类型。它与标准模式已经非常接近，现代开发里通常不需要刻意区分。

### 5.1. 一些不重要的细节

#### 准标准模式如何触发？

准标准模式由特定的旧式 `doctype` 触发。最常见的几种：

```html
<!-- HTML 4.01 Transitional -->
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">

<!-- XHTML 1.0 Transitional -->
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">
```

这类 `doctype` 有一个共同特征：它们引用了公共标识符，但「缺少 URI」（即后面没有跟上具体的 DTD 地址），或者使用了特定的 Transitional/Frameset 变体。

为什么叫「准标准」而不是「完全标准」？

准标准模式和标准模式在绝大多数场景下表现一致，唯一的实质差异在于表格中图片的垂直对齐方式：准标准模式下，表格单元格中的图片底部会留出一小段空白（受行内元素基线对齐影响），而标准模式下不会。这个差异只影响极少数布局场景，除此之外两者几乎没有区别。

今天的实际意义：如果你不写旧式 `doctype`，就不会触发准标准模式。现代页面只需写 `<!DOCTYPE html>`，直接进入标准模式，不需要关心它。

#### DTD 是什么？

DTD 全称是 Document Type Definition，即「文档类型定义」。它是 SGML/XML 体系下的一个概念，用来声明某个文档允许出现哪些元素、属性以及它们的嵌套规则。

用一个类比来理解：如果 HTML 文档是一份问卷，DTD 就是这份问卷的模板，规定了「哪些题目可以出现、选项该怎么填」。

在 HTML 语境下，DTD 的历史作用是：

早期 HTML（HTML 4.01 及之前）基于 SGML 体系，每个版本都有对应的 DTD 文件来定义合法的标签和属性。`doctype` 声明的原始目的就是指向这个 DTD 文件，告诉校验器和浏览器：「请按这个模板来检查我的页面」。

比如这段 `doctype`：

```html
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01 Transitional//EN" "http://www.w3.org/TR/html4/loose.dtd">
```

- `-//W3C//DTD HTML 4.01 Transitional//EN` 是公共标识符，相当于 DTD 的名字
- `http://www.w3.org/TR/html4/loose.dtd` 是 DTD 文件的实际地址

但到了 HTML5，DTD 被抛弃了。`<!DOCTYPE html>` 既不引用任何 DTD，也不基于 SGML。它唯一的作用就是触发标准模式。这是一个彻底的简化。

所以今天你只需要记住：DTD 是 `doctype` 的历史背景，现代 HTML 不再依赖它。

#### SGML 是什么？

SGML 全称是 Standard Generalized Markup Language，即「标准通用标记语言」。

它的核心思想很简单：用标签来描述文档的结构，用 DTD 来定义标签的规则。

可以把它理解成一个「标记语言的母语言」：

```
SGML（母语言，定义规则）
  ├── HTML 3.2（基于 SGML 的具体实现）
  ├── HTML 4.01（基于 SGML 的具体实现）
  ├── XHTML 1.0（基于 SGML 的具体实现）
  └── DocBook（基于 SGML 的技术文档标准）
  └── ...其他各种标记语言
```

SGML 不是某一种具体的标记语言，而是一套「元规则」 => 它规定了「怎么定义一种标记语言」。HTML 的早期版本（3.2、4.01 等）就是按照 SGML 的规则定义出来的具体标记语言。

历史脉络：

| 时间 | 发生了什么                                                  |
| ---- | ----------------------------------------------------------- |
| 1986 | SGML 成为 ISO 标准，主要用于出版和政府文档                  |
| 1991 | HTML 诞生，基于 SGML 的规则                                 |
| 1998 | XML 被设计为 SGML 的简化子集                                |
| 1999 | XHTML 1.0 诞生，基于 XML 的规则，比 SGML 时代的 HTML 更严格 |
| 2008 | HTML5 起草，明确脱离 SGML 体系，不再依赖 DTD                |

SGML 给 HTML 留下的最直接痕迹就是 `doctype`。早期的 `<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01//EN" ...>` 这种写法，就是 SGML 体系下的标准语法。而 HTML5 的 `<!DOCTYPE html>` 虽然保留了 `<!DOCTYPE` 这个形式，但本质上只是一个浏览器模式切换的开关，和 SGML 已经没有关系了。

## 6. 文档模式为什么会影响 JavaScript？

文档模式的主要差异看起来体现在 CSS 渲染上，但它会通过副作用影响 JavaScript。

原因并不神秘。JavaScript 读到的很多页面信息，本来就依赖浏览器如何解析和布局页面，例如：

- 元素尺寸和位置
- 盒模型相关结果
- 某些 DOM 行为和兼容性分支
- 不同浏览器在旧模式下暴露出的历史特性

也就是说，哪怕你写的 JavaScript 一模一样，只要页面处在不同文档模式下，最终观察到的运行结果也可能不一样。

所以后面如果笔记里提到某些 API、尺寸、事件或兼容问题，常常默认假设页面在标准模式下运行。这个前提要先立住。

## 7. 现在写页面时应该怎么做？

现代页面最稳妥的选择几乎总是：在文档开头写上 HTML5 的 `<!DOCTYPE html>`，让页面稳定运行在标准模式。

```html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Example</title>
  </head>
  <body></body>
</html>
```

这样做的意义主要有三点：

- 避免无意中落入怪异模式
- 让浏览器行为更接近现代标准预期
- 让后续 DOM、BOM、样式和布局问题更容易判断

如果你今天还碰到准标准模式或怪异模式，更多时候是在维护旧系统，而不是在设计新页面。对新项目来说，记住“始终显式声明 `doctype`”就够实用。
