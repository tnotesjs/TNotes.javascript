## 2. 🫧 评价

文档模式这件事很像老浏览器时代留下来的地基问题，平时不显眼，出问题时却能把布局和脚本一起带偏。把它记成“由 `doctype` 决定页面处于什么兼容语境”，后面排查怪异行为会省很多时间。

## 3. 🤔 文档模式是什么，为什么由 `doctype` 触发？

文档模式可以理解成浏览器解释页面时采用的兼容语境。它最早是 IE5.5 为了解决旧页面兼容问题引入的：同一个浏览器，需要既能尽量兼容旧网页，又能逐步支持更标准的行为，于是就有了模式切换。

这个切换通常由文档开头的 `doctype` 触发。浏览器看到不同的文档类型声明，会决定当前页面应该按哪一套规则去理解和渲染。

所以，`doctype` 不只是“写给校验器看的一行声明”，它还在告诉浏览器：这个页面应该按旧世界的兼容方式处理，还是按更标准的方式处理。

## 4. 🤔 混杂模式、标准模式、准标准模式有什么区别？

书里把文档模式分成三类：

| 模式 | 含义 | 典型特点 |
| --- | --- | --- |
| 混杂模式 | `quirks mode` | 更像早期浏览器的非标准行为，兼容历史页面 |
| 标准模式 | `standards mode` | 尽量按标准解释页面 |
| 准标准模式 | `almost standards mode` | 大体接近标准模式，只保留少数兼容差异 |

触发方式上，可以先记两个最重要的结论：

- 省略 `doctype`，通常会进入混杂模式
- 使用现代 HTML5 的 `<!DOCTYPE html>`，通常会进入标准模式

准标准模式主要是历史过渡产物，常见于某些旧的 HTML 4.01 / XHTML Transitional、Frameset 文档类型。它与标准模式已经非常接近，现代开发里通常不需要刻意区分。

## 5. 🤔 文档模式为什么会影响 JavaScript？

书里特别提醒了一点：文档模式的主要差异看起来体现在 CSS 渲染上，但它会通过副作用影响 JavaScript。

原因并不神秘。JavaScript 读到的很多页面信息，本来就依赖浏览器如何解析和布局页面，例如：

- 元素尺寸和位置
- 盒模型相关结果
- 某些 DOM 行为和兼容性分支
- 不同浏览器在旧模式下暴露出的历史特性

也就是说，哪怕你写的 JavaScript 一模一样，只要页面处在不同文档模式下，最终观察到的运行结果也可能不一样。

所以后面如果书里提到某些 API、尺寸、事件或兼容问题，常常默认假设页面在标准模式下运行。这个前提要先立住。

## 6. 🤔 现在写页面时应该怎么做？

现代页面最稳妥的选择几乎总是：在文档开头写上 HTML5 的 `<!DOCTYPE html>`，让页面稳定运行在标准模式。

```html
<!DOCTYPE html>
<html>
  <head>
    <meta charset="UTF-8" />
    <title>Example</title>
  </head>
  <body></body>
</html>
```

这样做的意义主要有三点：

- 避免无意中落入混杂模式
- 让浏览器行为更接近现代标准预期
- 让后续 DOM、BOM、样式和布局问题更容易判断

如果你今天还碰到准标准模式或混杂模式，更多时候是在维护旧系统，而不是在设计新页面。对新项目来说，记住“始终显式声明 `doctype`”就够实用。
