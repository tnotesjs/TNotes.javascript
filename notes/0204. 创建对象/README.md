## 2. 评价

- 这一节其实是在回答一个工程问题：如果你要批量创建很多结构类似的对象，应该把哪些东西放实例上，哪些东西放共享层。只会写 `new` 不够，你还得知道为什么方法放错地方会导致一堆重复函数对象。

## 3. 为什么“创建很多类似对象”会变成一个问题？

因为直接写对象字面量虽然简单，但一旦你要创建很多拥有同样接口的对象，就会不断重复同样的字段和方法定义。

```js
const person = {
  name: 'Nicholas',
  age: 29,
  sayName() {
    console.log(this.name)
  },
}
```

单个对象这样写没问题，但批量创建时就需要一种更可复用的模式。

## 4. 工厂模式和构造函数模式分别在做什么？

工厂模式的思路是：写一个函数，函数内部创建对象并返回它。

```js
function createPerson(name, age, job) {
  return {
    name,
    age,
    job,
    sayName() {
      console.log(this.name)
    },
  }
}
```

它解决了“重复创建类似对象”的问题，但没有很好回答“这个对象到底是什么类型”。

构造函数模式则是把创建逻辑挂到一个约定用 `new` 调用的函数上：

```js
function Person(name, age, job) {
  this.name = name
  this.age = age
  this.job = job
  this.sayName = function () {
    console.log(this.name)
  }
}
```

这样创建出来的实例就能和构造函数建立更明确的类型关系。

## 5. `new` 调用构造函数时到底发生了什么？

可以抓住这几步：

1. 创建一个新对象。
2. 把这个对象的原型指向构造函数的 `prototype`。
3. 让构造函数里的 `this` 指向这个新对象。
4. 执行构造函数代码，给新对象加属性。
5. 默认返回这个新对象。

这也是为什么下面这种写法能工作：

```js
const person = new Person('Nicholas', 29, 'Software Engineer')
```

同时也解释了为什么类构造函数和普通函数在调用方式上差异很大。

## 6. 构造函数模式为什么还不够好？

最大问题是：如果你把方法直接写在构造函数里，那么每创建一个实例，就会新建一份函数对象。

```js
function Person(name) {
  this.name = name
  this.sayName = function () {
    console.log(this.name)
  }
}
```

这会导致：

- 功能相同的方法无法共享
- 实例越多，重复函数越多

这也是为什么后面需要原型模式。

## 7. 原型模式为什么能解决方法复用问题？

因为原型是所有实例共享的一层对象，把方法放在原型上，所有实例都能访问到同一份实现。

```js
function Person() {}

Person.prototype.name = 'Nicholas'
Person.prototype.sayName = function () {
  console.log(this.name)
}
```

这样不同实例访问到的 `sayName` 实际上是同一个函数。

同时要记住属性查找顺序：

- 先找实例自身
- 再找原型对象
- 再沿原型链继续向上找

如果实例上定义了同名属性，它会遮蔽原型上的同名属性。

## 8. 创建对象时更稳定的实践是什么？

可以把经验收束成一句话：实例私有数据放构造函数里，共享方法放原型上。

这也是为什么在 `class` 出现之前，较稳定的做法通常是“构造函数 + 原型”组合。

如果只看本质，它解决的是两个目标同时成立：

- 每个实例都有自己的状态
- 多个实例共享同一套方法实现

理解这一点之后，再看后面的继承模式和 ES6 类，你会更容易明白它们到底在简化什么。
