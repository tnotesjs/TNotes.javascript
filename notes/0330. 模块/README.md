## 3. 🤔 为什么 JavaScript 需要模块？

现代 JavaScript 应用通常会包含大量文件、第三方库和业务功能。如果所有代码都放在全局作用域中，很容易出现命名冲突、依赖顺序混乱、隐式耦合和维护困难。

模块的核心作用，是把代码拆成独立的块：

- 每个模块有自己的内部实现。
- 模块可以决定哪些内容对外暴露。
- 模块可以声明自己依赖哪些外部模块。
- 模块系统负责连接这些模块，并按正确顺序加载和执行。

简单来说，模块化解决的是复杂性管理问题。它不是只为了拆文件，而是为了让代码之间的边界清楚、依赖明确、公共接口稳定。

## 4. 🤔 ES6 之前为什么会出现多种模块方案？

ES6 模块出现之前，浏览器和 ECMAScript 语言本身都没有原生模块机制。但真实项目又早就需要模块能力，于是社区只能用语言已有能力模拟模块。

常见路线包括：

| 方案 | 目标 | 特点 |
| --- | --- | --- |
| IIFE 模块模式 | 用函数作用域封装私有变量。 | 简单，但依赖顺序和加载需要手动管理。 |
| CommonJS | 服务端同步模块系统。 | 以 `require()` 和 `module.exports` 为核心，Node.js 生态广泛使用。 |
| AMD | 浏览器异步模块系统。 | 以 `define()` 声明依赖，适合按需异步加载。 |
| UMD | 兼容 CommonJS、AMD 和浏览器全局变量。 | 样板代码多，通常由构建工具生成。 |

这些方案都在试图表达同一件事：模块有依赖、有私有实现、有公共导出，只是受环境限制，表达方式不同。

## 5. 🤔 ES6 模块带来了什么变化？

ES6 把模块能力提升为语言规范的一部分。`import` 和 `export` 让模块依赖和公共接口可以被静态分析，浏览器也可以通过 `type="module"` 原生加载模块。

```js
// math.js
export function add(a, b) {
  return a + b
}
```

```js
// main.js
import { add } from './math.js'

console.log(add(1, 2))
```

和早期模块系统相比，ES6 模块的优势包括：

- 语法是标准的一部分，不依赖某个库约定。
- 依赖关系能被静态分析。
- 支持命名导出、默认导出和转移导出。
- 模块天然是严格模式。
- 模块作用域不会污染全局命名空间。
- 浏览器可以原生加载模块图。

## 6. 🤔 本章小结要抓住哪些结论？

模块模式是管理复杂性的长期工具。它让开发者可以创建彼此独立的代码段，在这些代码段之间声明依赖，并通过公共接口连接起来。

ES6 之前，CommonJS 和 AMD 分别面向服务器端同步加载与浏览器端异步加载。它们都很成功，但语法和运行模型不一致，也都需要额外工具才能在浏览器里顺畅使用。

ES6 模块把模块能力正式纳入语言，吸收了 CommonJS 和 AMD 的经验，用更简单的声明式语法表达依赖与导出。浏览器对原生模块的支持逐渐完善，构建工具也能帮助旧环境平滑过渡。
