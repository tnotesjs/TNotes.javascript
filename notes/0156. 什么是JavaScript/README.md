## 3. 🤔 JavaScript 最初要解决什么问题？

JavaScript 最初诞生在浏览器里，核心需求很朴素：把一些简单交互放到客户端执行，减少浏览器和服务器之间不必要的往返。

在 20 世纪 90 年代中期，很多用户通过调制解调器上网，网络速度很慢。一个表单如果只是在提交后才由服务器检查必填项是否为空，用户就要等待一次完整的请求和响应，体验很差。

JavaScript 最早的价值就在这里：

- 在页面提交前完成基础表单验证
- 在浏览器端响应用户操作
- 减少简单交互对服务器的依赖
- 让网页不只是静态文档

这也是为什么 JavaScript 一开始被称为客户端脚本语言。它并不是先以大型应用语言的形态出现，而是从解决网页交互中的具体痛点开始。

## 4. 🤔 为什么 JavaScript 后来不只是表单验证脚本？

JavaScript 的发展远远超出了最初的预期。它从表单验证脚本，逐步变成了浏览器中最重要的编程语言之一。

原因主要有三点：

- 浏览器成为应用运行平台，网页需要更复杂的交互。
- ECMAScript 标准持续演进，语言能力不断增强。
- DOM、BOM、网络请求、存储、媒体等 Web API 持续扩展。

今天的 JavaScript 可以处理复杂计算、页面交互、异步任务、模块组织、前端工程化，甚至可以通过 Node.js 运行在服务器端。它的使用场景已经不再局限于浏览器窗口中的一点脚本。

不过，也正因为它应用范围很广，学习 JavaScript 时要先分清两个层次：

- 语言本身提供什么能力
- 运行环境额外提供什么能力

例如：

- `let`、`Promise`、`Array` 属于语言层面的内容
- `document`、`window`、`fetch`、`localStorage` 则来自具体宿主环境或 Web 标准

## 5. 🤔 JavaScript 和宿主环境是什么关系？

JavaScript 代码必须运行在某个宿主环境中。宿主环境负责提供语言执行所需的上下文，也负责暴露和外部世界交互的能力。

常见宿主环境包括：

| 宿主环境       | 典型能力                                 |
| -------------- | ---------------------------------------- |
| 浏览器         | DOM、BOM、事件、网络请求、存储、媒体能力 |
| Node.js        | 文件系统、进程、网络服务、模块加载       |
| 嵌入式运行环境 | 设备控制、脚本扩展、自动化能力           |

ECMAScript 标准本身并不定义文件系统、网络请求、页面结构或用户界面。它定义的是语言核心，比如语法、类型、对象、语句、操作符和全局对象等。

所以，当你在浏览器里写 `document.querySelector()` 时，你使用的并不是 ECMAScript 语言核心，而是浏览器提供的 DOM API。当你在 Node.js 里使用 `fs.readFile()` 时，你使用的是 Node.js 提供的宿主能力。

## 6. 🤔 学习 JavaScript 为什么要先理解它的边界？

JavaScript 的学习容易混在一起：语法、浏览器 API、工程工具、框架写法经常同时出现。如果一开始不分层，就很容易把不同来源的能力混为一谈。

一个更稳定的理解方式是：

| 层次 | 关注点 | 示例 |
| --- | --- | --- |
| ECMAScript | 语言核心 | 类型、变量、函数、类、模块、期约 |
| DOM | 页面文档 | 元素查询、节点操作、事件绑定 |
| BOM 和 Web API | 浏览器环境 | `window`、`location`、`history`、`fetch`、存储 |
| 工程生态 | 开发组织方式 | 打包、转译、测试、框架 |

后续学习中，很多问题都可以先问一句：这是语言能力，还是宿主环境能力？

这个问题看似简单，但它能帮助你更准确地理解兼容性、标准来源和运行环境差异。

## 7. 🤔 这一章的笔记主要介绍什么内容？

这一章可以压缩成一句话：JavaScript 是从浏览器客户端脚本发展而来的语言，它的完整浏览器实现由 ECMAScript、DOM 和 BOM 共同构成。

更细一点，可以分成三条主线：

| 主线 | 结论                                             |
| ---- | ------------------------------------------------ |
| 历史 | JavaScript 起源于浏览器端表单验证和页面交互需求  |
| 标准 | ECMAScript 提供语言核心，ECMA-262 是主要规范依据 |
| 环境 | DOM 和 BOM 让 JavaScript 可以操作网页和浏览器    |

这三条线会贯穿后续学习。语言基础看 ECMAScript，页面操作看 DOM，浏览器窗口和导航能力看 BOM。

版本问题也可以按这个思路梳理：

- 如果讨论语言特性，优先看 ECMAScript 标准版本
- 如果讨论浏览器行为，继续看具体浏览器和 Web 标准实现
- 如果讨论历史兼容性，再回到 JavaScript 1.x、JScript 或旧浏览器差异（现代开发中几乎不会遇到）

后续章节会进入语法、类型、对象、DOM、BOM、事件、网络请求、模块和工作者线程等具体主题。学习时可以一直带着三个问题：

- 这个能力来自 ECMAScript，还是来自浏览器环境？
- 这个能力是否有标准定义，标准由谁维护？
- 这个能力在目标运行环境中的支持情况如何？

这三个问题能帮助你避免把语言、平台和工具混在一起，也能让你在遇到兼容性问题时更快定位原因。
