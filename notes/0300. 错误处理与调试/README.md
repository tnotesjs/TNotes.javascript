## 3. 🤔 为什么错误处理和调试要放在一起理解？

JavaScript 很长一段时间都被认为难调试。原因并不只是语言动态，而是早期浏览器给出的错误信息非常有限，错误消息经常只有一句模糊描述，缺少调用栈、源码位置和上下文。

随着 `try/catch`、`throw`、内置错误类型、开发者工具和浏览器控制台逐渐成熟，前端开发者才有了比较完整的排错链路。

这条链路可以拆成三步：

1. 发现错误：浏览器控制台、全局错误事件、日志系统。
2. 处理错误：`try/catch`、自定义错误、降级策略。
3. 定位错误：控制台日志、断点、运行时检查、错误上报。

调试关注的是怎么找到问题，错误处理关注的是问题发生后系统怎么继续工作。两者配合起来，才能把错误从随机崩溃变成可观察、可恢复、可修复的问题。

## 4. 🤔 浏览器默认会如何报告错误？

现代桌面浏览器都会在开发者工具的控制台中报告 JavaScript 错误。开发时，你通常可以通过右键检查、快捷键或浏览器菜单打开开发者工具，再切到 Console 面板查看错误。

移动端通常不会直接在设备上提供完整控制台，更常见的方式是通过远程调试把手机浏览器连接到桌面浏览器中查看日志和错误。

不过，浏览器默认错误报告主要服务开发者。普通用户通常看不到这些信息，也看不懂这些信息。因此，生产环境不能只依赖浏览器控制台，而应该有应用自己的错误提示、恢复策略和上报机制。

## 5. 🤔 JavaScript 提供了哪些错误处理机制？

核心语言机制包括：

| 机制             | 作用                                                      |
| ---------------- | --------------------------------------------------------- |
| `try/catch`      | 捕获同步代码中抛出的异常。                                |
| `finally`        | 无论是否出错都执行清理逻辑。                              |
| `throw`          | 主动抛出错误。                                            |
| 内置错误类型     | 区分 `TypeError`、`ReferenceError`、`RangeError` 等错误。 |
| 自定义错误类型   | 为应用或库表达更明确的错误语义。                          |
| `window.onerror` | 捕获未处理运行时错误的兜底入口。                          |

这些机制不是互相替代的关系。底层函数通常负责验证输入并抛出清晰错误；上层业务逻辑在知道如何恢复时再捕获错误；全局错误处理只作为最后一道防线。

## 6. 🤔 错误处理策略应该关注什么？

好的错误处理策略不只是写几个 `catch`。它至少要回答几个问题：

- 哪些错误可能发生？
- 哪些错误可以提前用类型检查、参数校验、静态分析避免？
- 哪些错误只影响局部功能？
- 哪些错误会让用户的主要任务无法继续？
- 错误发生后要不要提示用户？
- 错误是否要记录到服务器，方便后续分析？

对用户来说，最糟糕的情况不是页面出现技术错误，而是页面悄悄坏掉、没有解释、也没有恢复方式。错误处理的目标，是让用户尽可能继续完成目标，并让开发者有足够信息修复根因。

## 7. 🤔 本章小结要抓住哪些结论？

这一章强调了复杂 Web 应用中错误处理的重要性。JavaScript 错误如果没有被预测、捕获和恢复，可能会直接破坏用户体验。

可以用 `try/catch` 捕获已知风险代码，用 `throw` 抛出清晰的自定义错误，用 `window.onerror` 兜底未处理错误，并把关键错误上报到服务器。除此之外，还要通过严格比较、类型检查、参数验证、查询字符串编码和静态分析减少错误发生的机会。

调试方面，现代浏览器提供了控制台、断点、运行时检查和 `debugger` 语句。相比早期依赖 `alert()` 的方式，现在更推荐用控制台、断点和日志系统定位问题。

旧版 IE 的一些错误现在更多是历史知识，但它们提醒你：浏览器错误消息并不总是直接说明根因。真正可靠的做法，是让自己的代码提供足够明确的上下文。
