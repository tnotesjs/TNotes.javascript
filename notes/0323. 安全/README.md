## 2. 🫧 评价

## 3. 🤔 为什么 Ajax URL 必须像普通接口一样做授权？

Ajax 请求只是由 JavaScript 发起的 HTTP 请求。任何 Ajax 能访问的 URL，理论上都可能被用户直接访问、被脚本构造访问，或被其他页面诱导访问。

例如：

```txt
/get-user-info?id=23
```

如果用户把 `23` 改成 `24`，服务器必须判断当前用户是否有权访问 ID 为 `24` 的数据。不能因为接口原本只打算给页面脚本调用，就跳过权限校验。

前端隐藏按钮、隐藏 URL、混淆参数名，都不是安全边界。真正的安全判断必须在服务端完成。

## 4. 🤔 CSRF 为什么会影响 Ajax 应用？

CSRF 是跨站请求伪造。攻击页面诱导用户浏览器向受信任站点发请求，而浏览器可能自动带上该站点的 Cookie。

如果服务端只根据 Cookie 判断用户身份，却不确认请求是否由合法页面发起，就可能执行攻击者构造的操作。

例如，攻击页面可以诱导浏览器请求某个转账、删除、修改配置的接口。如果接口没有额外校验，就可能产生安全问题。

## 5. 🤔 哪些防护手段更可靠？

常见防护包括：

- 服务端对每个接口做身份认证和权限校验。
- 使用 HTTPS，避免请求内容和 token 被窃听或篡改。
- 对状态改变请求使用 CSRF token。
- 设置合理的 SameSite Cookie 策略。
- 对关键请求校验 `Origin` 或 `Referer`，作为辅助信号。
- 对敏感操作增加二次确认、短期令牌或重放保护。

示例：

```js
await fetch('/api/transfer', {
  method: 'POST',
  credentials: 'same-origin',
  headers: {
    'Content-Type': 'application/x-www-form-urlencoded',
    'X-CSRF-Token': csrfToken,
  },
  body: new URLSearchParams({
    to: 'user-123',
    amount: '100',
  }),
})
```

这里的 token 需要由服务端生成和校验，不能只是前端自己写一个固定值。

## 6. 🤔 哪些做法不足以防护 CSRF？

下面这些手段单独使用都不可靠：

- 只要求使用 POST。攻击者也可能构造 POST 请求。
- 只依赖 Cookie。Cookie 可能被浏览器自动携带。
- 只在前端隐藏接口入口。攻击者可以直接构造请求。
- 只检查参数格式。格式正确不代表用户有权限。

`Origin` 和 `Referer` 可以作为辅助校验，但不能替代服务端授权和 CSRF token。它们可能受隐私策略、代理、浏览器配置等因素影响。

## 7. 🤔 CORS 能不能当作安全认证？

CORS 不是认证授权机制。它只决定浏览器是否允许某个页面脚本读取跨源响应。

即使 CORS 没有放行，服务器也可能已经收到了请求。对于会改变状态的接口，服务端仍然要校验用户身份、权限、CSRF token 和请求参数。

反过来，即使 CORS 放行了某个来源，也不代表这个来源里的所有用户都有权访问所有数据。业务授权仍然必须在服务端完成。

## 8. 🤔 前端网络请求安全的基本原则是什么？

可以抓住几条原则：

- 所有接口都按公开入口设计。
- 服务端永远重新验证身份和权限。
- 前端传来的任何数据都不可信。
- Cookie 自动携带不等于用户主动授权。
- CORS 是浏览器读取策略，不是业务权限系统。
- 敏感操作使用 HTTPS、CSRF token 和合理的 Cookie 策略。

前端可以帮助用户正确发请求，但不能替服务端做最终安全判断。
