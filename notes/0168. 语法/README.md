## 2. 🫧 评价

语法规则不难，但它很重要，因为它决定了代码是否能被正确解析。

除了「严格模式」之外的其它知识点，基本上也是其它编程语言的基础语法规则，这些知识点是通用的，掌握了这些规则，学习其它编程语言也会更快上手。

- 语句结尾稳定

## 3. 🤔 大小写在 ECMAScript 中有什么影响？

ECMAScript 中的变量名、函数名、属性名、关键字和操作符都区分大小写。

这意味着 `test` 和 `Test` 是两个不同的标识符：

```js
let test = 1
let Test = 2

console.log(test) // 1
console.log(Test) // 2
```

同理，`typeof` 是关键字，不能拿来当函数名；但 `Typeof` 只是普通标识符，从语法上看是可以使用的。

实际写代码时，大小写差异最好只用于明确的命名约定，不要用来制造两个容易混淆的变量名。否则阅读和调试都会很吃力。

## 5. 🤔 转义符是什么？有什么用？

### 5.1. 转义符是什么

先来看维基百科对转义符的介绍 => [维基百科 - Escape character][1]

In computing and telecommunication, an escape character is a character that invokes an alternative interpretation on the following characters in a character sequence. An escape character is a particular case of metacharacters. Generally, the judgement of whether something is an escape character or not depends on the context.

中文：在计算机和通信领域中，转义字符是一种字符，它会对其后的字符序列中的字符进行替代解释。转义字符是元字符的一种特殊情况。通常，判断某个字符是否为转义字符取决于上下文环境。

### 5.2. 转义符有什么用

转义字符是编程语言中一种非常重要的机制，它允许开发者在字符串中安全地使用特殊字符。有些字符在字符串文本中有特殊的含义，例如：

- 引号：
  - `
  - `'`
  - `"`
- 换行符：`\n`
- 制表符：`\t`
- ……

如果你想在字符串中包含这些特殊字符，而不是它们的特殊功能，就需要一种方法来告诉编程语言这些字符应该被视为普通文本。

转义字符允许你插入这些特殊字符而不触发它们的特殊行为。

| 转义符 | 含义           |
| ------ | -------------- |
| `\'`   | 普通英文单引号 |
| `\"`   | 普通英文双引号 |
| `\r`   | 回车           |
| `\n`   | 换行           |
| ……     | ……             |

### 5.3. 转义符的英文为啥叫 `escape character`

`escape` 表示逃离的意思，`escape character` 表示转义字符（可以理解为逃离计算机的翻译）。

在编程中，转义符的作用是让某些字符“逃脱”其正常的解释方式，从而被解释为特殊含义或符号。这种机制与“逃脱”或“避开”字符的原始含义非常相似，因此“escape”这个词非常贴切。

在阅读外文文档的时候，需要知道这个短语 `escape character` 表示的是转义符的含义，知道这一点能帮助咱们更好地查阅资料。

### 5.4. 转义符 `\r\n` 诞生的背景 - 旧版打字机【不重要】

![图 0](https://cdn.jsdelivr.net/gh/tnotesjs/imgs@main/2025-08-19-21-52-20.png)

`\r\n` 的诞生背景，和历史上的打字机有关。

- 回车换行 `\r\n`
- 回车 `\r`
- 换行 `\n`
- `\r\n` 的由来与历史上的 打字机 和早期计算机终端的工作方式有关。理解这些字符的背景，需要回顾计算机文本处理的早期历史。
- `\r\n` 的使用是一个从早期打字机操作演变而来的历史遗留问题，它体现了计算机技术发展过程中的连续性和兼容性考虑。

打字机的工作原理

- 在早期的打字机上，当打字员达到一行的末尾时，需要执行两个动作来开始新的一行：
  1. 回车（Carriage Return, CR） - 这个操作使打字机的打印头回到左边界的起始位置。
  2. 换行（Line Feed, LF） - 这个操作使纸张向上滚动一行。
- 这 两个动作 分别对应于 `\r` （回车符）和 `\n` （换行符）。

电传打字机和早期计算机

- 随着电传打字机（Teletypes）和早期计算机终端的使用，这些设备也采用了类似的机械操作方式。
- 在这些设备中，`CR` 和 `LF` 同样是控制字符，用于文本的格式化和定位。

操作系统

- 随着计算机操作系统的发展，不同系统采纳了不同的约定来表示文本中的行结束。
- Unix 和后来的 Linux 及 macOS：
  - 采用简单的 `\n` 来表示行结束，这可能是因为它简化了处理逻辑，并且节省了存储空间。
- Windows（及之前的 DOS）：
  - 采用 `\r\n` 组合来模仿早期电传打字机的行为，保持与之前技术的兼容性。
  - 在这些系统中，`\r\n` 确保了文本在编辑和显示时的一致性，尤其是在涉及多种设备和上下文（如网络传输和文本文件处理）时。

评论

- 从这个历史小片段来看，其实也反映出了 macOS 和 Windows 系统在设计上的一些策略：
- Windows：保留技术债 - 比如在做系统升级（比如 win10 -> win11）的时候，会尽可能去考虑之前软件的兼容性。对于早期系统设计上一些不太合理的地方，从技术角度来看是可以优化处理掉的，可是却没有这么做，一些历史问题被保留了一下。
- macOS：舍弃技术债 - 系统一升级，很多旧版软件可能压根就打不开了。

现代应用

- 尽管现代计算机和打印技术不再需要物理的“回车”和“换行”操作，`\r\n` 作为行结束符的约定仍然在 Windows 系统中保留。这种历史遗留的特性意味着跨平台软件开发和网络通信协议（如 HTTP 和电子邮件标准）中仍需考虑不同的行结束符处理。

## 6. 🤔 标识符应该怎样命名？

标识符就是变量、函数、属性或参数的名称。

基本规则是：

- 第一个字符必须是字母、下划线 `_` 或美元符号 `$`。
- 后续字符可以是字母、下划线、美元符号或数字。
- 不能使用关键字、保留字、`true`、`false` 和 `null`。

ECMAScript 允许在标识符里使用某些扩展字符和 Unicode 字符，但日常开发中不推荐这样做。

```js
let π = 3.14159
console.log(π * 2) // 6.28318

let 用户名 = 'abc'
console.log(用户名) // 'abc'
```

最常见、也最容易协作的风格是驼峰命名：

```js
let firstName = 'Ada'
let currentUser = { name: 'Ada' }

function doSomethingImportant() {}
```

命名本身不是语法功能，但好的命名能降低代码理解成本。

## 7. 🤔 注释和严格模式怎么用？

### 7.1. 注释

ECMAScript 支持两种 C 风格注释。

单行注释使用 `//`：

```js
// 这是单行注释
const count = 10
```

块注释使用 `/* ... */`：

```js
/*
  这是块注释，
  可以跨多行。
*/
const count = 10
```

### 7.2. 严格模式

严格模式是 ES5 引入的一种更严格的解析和执行模型。它会让一些不规范、不安全或容易出错的写法直接抛错。

启用整个脚本的严格模式：

```js
'use strict'
```

也可以只在函数内部启用：

```js
function doSomething() {
  'use strict'
  // 函数体
}
```

严格模式的写法看起来像普通字符串，但 JavaScript 引擎会把它识别为指令。

#### 我很少能在开源项目中看到显式写 'use strict' 的了，现代开发还用显式开启严格模式吗？

好问题。答案是：在现代开发中，大多数场景下不需要显式写了，因为严格模式已经被自动启用。

ES 模块自带严格模式：

```js
// 只要使用 import 或 export，严格模式自动生效
import { foo } from './foo.js'
export function bar() {}
```

Vite、Webpack 等构建工具打包后的产物通常也是模块代码，所以严格模式自动覆盖了整个项目。这就是你很少看到 `'use strict'` 的根本原因。

类体内部也自带严格模式：

```js
class User {
  constructor() {
    // 这里自动处于严格模式
  }
}
```

还可能需要显式写 `'use strict'` 的场景：

```js
// CommonJS 模块（Node.js 传统写法）
'use strict'
const fs = require('fs')
```

如果项目仍然使用 CommonJS（没有 `type: "module"` 且没有构建工具），那 `'use strict'` 还是需要手动声明的。

所以总结一下：`'use strict'` 不是过时了，而是被现代模块系统接管了。你不需要手动写它，前提是你的项目在使用 ES 模块或类语法。

## 8. 🤔 在使用分号和代码块 `{}` 时需要注意什么？

### 8.1. 分号

ECMAScript 语句通常以分号结尾。分号在很多场景下可以省略，因为解析器会尝试通过自动分号插入（ASI）机制来补全语句结尾。

但 ASI 并不总是符合你的预期：

```js
function getData() {
  return
  {
    name: 'Ada'
  }
}

console.log(getData()) // undefined
```

这里你本来可能是想返回一个对象，但 ASI 会在 `return` 后面自动插入分号，导致函数实际返回的是 `undefined`。

这类问题的隐蔽之处在于：代码看起来没有语法错误，但运行结果和预期不同。因此实践中建议保持一致的风格 => 无论团队选择加分号还是不加分号，都应该让格式化工具稳定处理，而不是依赖 ASI 的默认行为。

### 8.2. 代码块

和分号一样，代码块的使用也有一个「能不能省」和「该不该省」的区别。

代码块由 `{}` 包裹，用来把多条语句组合在一起。某些控制语句在只有一条语句时可以省略代码块，但不推荐：

```js
// 不推荐
if (enabled) console.log('enabled')

// 推荐
if (enabled) {
  console.log('enabled')
}
```

始终使用代码块，能让控制结构更清楚，也能减少后续插入新语句时引入错误的概率。

::: tip

在刷算法题时，如果流程控制语句（如：条件语句、循环语句等）只有一行，很多人往往喜欢省略代码块来节省行数，以起到视觉上的简洁效果，这一点感觉无可厚非，我也经常这么做！除了算法题之外，在实战项目中我也经常这么干！但在实战项目中，养成总是使用代码块的习惯，相对而言还是是会更加保险一些的。

:::
