## 2. 🫧 评价

语法规则不难，但它很重要，因为它决定了代码是否能被正确解析。

除了「严格模式」之外的其它知识点，基本上也是其它编程语言的基础语法规则，这些知识点是通用的，掌握了这些规则，学习其它编程语言也会更快上手。

值得养成几点习惯：

- 标识符命名清楚
- 语句结尾稳定
- 控制结构总是加代码块

## 3. 🤔 大小写在 ECMAScript 中有什么影响？

ECMAScript 中的变量名、函数名、属性名、关键字和操作符都区分大小写。

这意味着 `test` 和 `Test` 是两个不同的标识符：

```js
let test = 1
let Test = 2

console.log(test) // 1
console.log(Test) // 2
```

同理，`typeof` 是关键字，不能拿来当函数名；但 `Typeof` 只是普通标识符，从语法上看是可以使用的。

实际写代码时，大小写差异最好只用于明确的命名约定，不要用来制造两个容易混淆的变量名。否则阅读和调试都会很吃力。

## 4. 🤔 标识符应该怎样命名？

标识符就是变量、函数、属性或参数的名称。

基本规则是：

- 第一个字符必须是字母、下划线 `_` 或美元符号 `$`。
- 后续字符可以是字母、下划线、美元符号或数字。
- 不能使用关键字、保留字、`true`、`false` 和 `null`。

ECMAScript 允许在标识符里使用某些扩展字符和 Unicode 字符，但日常开发中不推荐这样做。

```js
let π = 3.14159
console.log(π * 2) // 6.28318

let 用户名 = 'abc'
console.log(用户名) // 'abc'
```

最常见、也最容易协作的风格是驼峰命名：

```js
let firstName = 'Ada'
let currentUser = { name: 'Ada' }

function doSomethingImportant() {}
```

命名本身不是语法功能，但好的命名能降低代码理解成本。

## 5. 🤔 注释和严格模式怎么用？

### 5.1. 注释

ECMAScript 支持两种 C 风格注释。

单行注释使用 `//`：

```js
// 这是单行注释
const count = 10
```

块注释使用 `/* ... */`：

```js
/*
  这是块注释，
  可以跨多行。
*/
const count = 10
```

### 5.2. 严格模式

严格模式是 ES5 引入的一种更严格的解析和执行模型。它会让一些不规范、不安全或容易出错的写法直接抛错。

启用整个脚本的严格模式：

```js
'use strict'
```

也可以只在函数内部启用：

```js
function doSomething() {
  'use strict'
  // 函数体
}
```

严格模式的写法看起来像普通字符串，但 JavaScript 引擎会把它识别为指令。

#### 我很少能在开源项目中看到显式写 'use strict' 的了，现代开发还用显式开启严格模式吗？

好问题。答案是：在现代开发中，大多数场景下不需要显式写了，因为严格模式已经被自动启用。

ES 模块自带严格模式：

```js
// 只要使用 import 或 export，严格模式自动生效
import { foo } from './foo.js'
export function bar() {}
```

Vite、Webpack 等构建工具打包后的产物通常也是模块代码，所以严格模式自动覆盖了整个项目。这就是你很少看到 `'use strict'` 的根本原因。

类体内部也自带严格模式：

```js
class User {
  constructor() {
    // 这里自动处于严格模式
  }
}
```

还可能需要显式写 `'use strict'` 的场景：

```js
// CommonJS 模块（Node.js 传统写法）
'use strict'
const fs = require('fs')
```

如果项目仍然使用 CommonJS（没有 `type: "module"` 且没有构建工具），那 `'use strict'` 还是需要手动声明的。

所以总结一下：`'use strict'` 不是过时了，而是被现代模块系统接管了。你不需要手动写它，前提是你的项目在使用 ES 模块或类语法。

## 6. 🤔 在使用分号和代码块 `{}` 时需要注意什么？

### 6.1. 分号

ECMAScript 语句通常以分号结尾。分号在很多场景下可以省略，因为解析器会尝试通过自动分号插入（ASI）机制来补全语句结尾。

但 ASI 并不总是符合你的预期：

```js
function getData() {
  return
  {
    name: 'Ada'
  }
}

console.log(getData()) // undefined
```

这里你本来可能是想返回一个对象，但 ASI 会在 `return` 后面自动插入分号，导致函数实际返回的是 `undefined`。

这类问题的隐蔽之处在于：代码看起来没有语法错误，但运行结果和预期不同。因此实践中建议保持一致的风格 => 无论团队选择加分号还是不加分号，都应该让格式化工具稳定处理，而不是依赖 ASI 的默认行为。

### 6.2. 代码块

和分号一样，代码块的使用也有一个「能不能省」和「该不该省」的区别。

代码块由 `{}` 包裹，用来把多条语句组合在一起。某些控制语句在只有一条语句时可以省略代码块，但不推荐：

```js
// 不推荐
if (enabled) console.log('enabled')

// 推荐
if (enabled) {
  console.log('enabled')
}
```

始终使用代码块，能让控制结构更清楚，也能减少后续插入新语句时引入错误的概率。

::: tip

在刷算法题时，如果流程控制语句（如：条件语句、循环语句等）只有一行，很多人往往喜欢省略代码块来节省行数，以起到视觉上的简洁效果，这一点感觉无可厚非，我也经常这么做！除了算法题之外，在实战项目中我也经常这么干！但在实战项目中，养成总是使用代码块的习惯，相对而言还是是会更加保险一些的。

:::
