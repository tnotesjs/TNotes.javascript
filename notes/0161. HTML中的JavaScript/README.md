## 3. 🤔 把 JavaScript 放进 HTML 页面时，核心问题是什么？

JavaScript 要在网页里工作，第一件事不是讨论语法，而是先回答一个更基础的问题：它怎样和 HTML 一起被浏览器解析，而且不把页面本身搞乱。

网页的主导语言是 HTML。浏览器会一边下载文档，一边从上到下解析标记；而 JavaScript 一旦插入这个过程中，就可能打断解析、修改 DOM、依赖前面的脚本结果，甚至在脚本不可用时影响页面可用性。

所以这一章真正要建立的是一套“页面级规则”：

- JavaScript 通过什么入口接入 HTML
- 浏览器按什么顺序下载和执行脚本
- 脚本会不会阻塞页面渲染
- 页面处于什么文档模式
- 脚本不可用时用户还能看到什么

这些问题看起来像外围细节，但它们决定了脚本是否稳定、页面是否可读、性能是否可接受。

## 4. 🤔 `<script>` 在网页中承担什么角色？

JavaScript 被放进 HTML 页面，主要依靠 `<script>` 元素。它既可以直接包裹一段行内代码，也可以通过 `src` 属性去加载外部脚本文件。

简单来说，`<script>` 做了三件事：

- 告诉浏览器这里开始出现脚本内容
- 指定脚本来自当前文档还是外部文件
- 允许开发者声明脚本应该怎样下载和执行

默认情况下，浏览器会按照 `<script>` 在文档中出现的顺序解释它们。也就是说，后一个脚本往往依赖前一个脚本已经执行完成，这就是为什么脚本顺序经常不是“看起来差不多就行”，而是有明确语义的。

另外，`src` 可以指向同源资源，也可以指向别的域名。浏览器允许发起这类请求，但你仍然要谨慎：被加载进来的外部脚本会以当前页面的一部分身份运行，所以来源必须可信。

## 5. 🤔 脚本加载方式为什么会直接影响页面体验？

脚本最让人头疼的一点是：默认会阻塞。

不管是行内脚本，还是普通的外部脚本，只要浏览器解析到它，就要先停下来处理脚本，然后才能继续解析后面的 HTML。对用户来说，这意味着：如果脚本放得太早、太多，页面就更容易长时间白屏。

因此，本章围绕脚本加载给出了一组很实用的经验：

- 不依赖提前执行的脚本，通常放在 `</body>` 前更合适
- 需要保留执行顺序、又不想阻塞解析时，可以考虑 `defer`
- 没有相互依赖、允许先下载先执行的脚本，可以考虑 `async`
- 如果你用 DOM 动态插入脚本，还要额外考虑默认异步和预加载不可见的问题

也就是说，脚本不仅是“要不要写”的问题，更是“什么时候下载、什么时候执行、会不会打断页面”的问题。

## 6. 🤔 浏览器环境里还要考虑哪些“页面级约束”？

除了脚本本身，这一章还提醒你注意两个经常被忽略的页面级约束：文档模式和无脚本兜底。

- 第一个约束是文档模式。浏览器会根据 `doctype` 判断当前页面处于混杂模式、标准模式还是准标准模式。它们的主要差异常常体现在布局和渲染上，但这些差异也会以副作用的方式影响 JavaScript 观察到的结果，比如尺寸、节点行为或兼容性表现。
- 第二个约束是脚本不可用时的用户体验。虽然现代浏览器基本都支持 JavaScript，但用户依然可能禁用脚本，或者因为策略、插件、网络环境让某些脚本没有正常运行。`<noscript>` 的价值就在这里：至少给用户一个明确说明，或者提供最低限度的替代内容。

换句话说，JavaScript 在网页里不是孤立运行的，它始终受 HTML 解析规则、浏览器模式和页面降级策略共同约束。

## 7. 🤔 这一章都有哪些核心内容？

这一章可以总结为一句话：JavaScript 进入网页，不只是“把代码写进去”，而是要按照浏览器的页面解析规则被加载、执行和兜底。

把重点压缩一下，就是下面这几条：

- JavaScript 通过 `<script>` 元素插入 HTML 页面
- 默认脚本按出现顺序执行，并且会阻塞后续解析
- `defer` 适合需要保留顺序的延迟执行脚本，`async` 适合彼此独立的异步脚本
- 行内脚本不是不能用，但外部文件通常更利于维护、缓存和长期演进
- 文档模式会影响浏览器行为，现代页面应尽量稳定在标准模式
- `<noscript>` 用于脚本不可用时的替代内容

后面学习 DOM、事件、BOM 和浏览器 API 时，你可以一直带着一个判断框架：这个问题到底是语言本身的问题，还是脚本嵌入网页后的加载与运行问题。很多“为什么页面行为不对”的问题，答案都藏在这一章。
