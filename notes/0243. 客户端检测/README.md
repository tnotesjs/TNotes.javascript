## 2. 🫧 评价

- 客户端检测不是炫技入口，而是兼容性兜底手段。最好的检测策略通常是先写通用方案，再用最小范围的检测修补真实差异。

## 3. 🤔 为什么需要客户端检测？

理想情况下，所有浏览器都应该支持同一套标准接口，并且表现一致。但现实中，不同浏览器、不同版本、不同操作系统和设备之间仍然会有差异。

客户端检测就是为这些差异准备的补救手段。它帮助你判断当前运行环境是否支持某个能力，或者当前环境大概属于哪类浏览器、系统和设备。

不过，客户端检测应该是最后手段。能用标准 API、渐进增强、CSS 响应式和通用逻辑解决的问题，不应该先写浏览器分支。

## 4. 🤔 常见客户端检测方式有哪些？

本章主要讨论三类检测：

| 检测方式 | 关注点 | 推荐程度 |
| --- | --- | --- |
| 能力检测 | 当前环境是否支持某个具体特性 | 首选 |
| 用户代理检测 | 通过 `navigator.userAgent` 推断浏览器、系统和设备 | 兜底 |
| 软件与硬件检测 | 读取 `navigator`、`screen` 等对象暴露的环境信息 | 按需使用 |

能力检测通常最可靠，因为它不关心浏览器叫什么，只关心你要用的能力是否存在。

用户代理检测可以推断浏览器身份，但用户代理字符串历史包袱很重，也可能被伪造或弱化。

软件与硬件检测能提供设备能力、网络、电池、定位等信息，但支持度和隐私限制差异较大。

## 5. 🤔 客户端检测应该怎么选？

可以按这个顺序考虑：

1. 先设计不依赖特定浏览器的通用方案。
2. 如果必须分支，优先检测需要使用的具体能力。
3. 如果能力检测不能表达需求，再考虑读取软件或硬件信息。
4. 只有在没有更好选择时，才使用用户代理检测。

例如，如果你想判断能不能使用地理定位，不应该先判断是不是某个浏览器，而应该直接判断 `navigator.geolocation` 是否存在。

```js
if ('geolocation' in navigator) {
  navigator.geolocation.getCurrentPosition(console.log, console.error)
}
```

这段代码回答的是“能不能用定位”，而不是“这是不是某个浏览器”。这通常就是更稳的检测思路。

## 6. 🤔 学完这一章应该留下哪些结论？

本章可以收束为几条实用结论：

- 客户端检测是兼容性补救，不是默认架构策略。
- 能力检测应该优先用于决定代码下一步怎么执行。
- 能力存在不等于行为符合预期，必要时要检测类型和行为。
- 通过能力检测推断浏览器身份有局限，不能把它当作稳定身份标识。
- 用户代理检测历史复杂，字符串里有大量兼容性伪装信息。
- 用户代理检测可以识别浏览器、渲染引擎、系统和设备，但需要持续维护解析规则。
- `navigator` 和 `screen` 能暴露软件硬件信息，但这些 API 常受浏览器支持、权限、安全上下文和隐私策略影响。

简单来说，客户端检测的核心原则是：检测你真正要用的能力，而不是先给浏览器贴标签。
