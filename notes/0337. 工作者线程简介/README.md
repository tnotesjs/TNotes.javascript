## 2. 🫧 评价

- 学工作者线程时，先把它当成独立的 JavaScript 沙盒会更顺：它能并行做事，但不能像普通线程那样随意共享页面内存和 DOM。

## 3. 🤔 工作者线程运行在什么环境里？

浏览器中的每个页面都可以看作一个独立环境：它有自己的内存、事件循环、DOM、全局对象和一组浏览器 API。不同页面之间通常彼此隔离，浏览器负责并行管理这些环境。

工作者线程是在页面环境之外创建的二级环境。它可以并行执行 JavaScript，但不能直接使用依赖页面主线程的 API，例如 DOM。

换句话说，工作者线程不是页面主线程的一个函数调用栈，而是另一个独立执行上下文。它和主线程之间通过消息沟通。

```js
const worker = new Worker('./worker.js')

worker.postMessage('hello')
worker.onmessage = (event) => {
  console.log(event.data)
}
```

## 4. 🤔 工作者线程和传统线程有什么相似点？

工作者线程和传统线程有不少相似之处。

| 相似点 | 说明 |
| --- | --- |
| 可能由底层真实线程实现 | 浏览器引擎通常会把工作者线程映射到底层线程或线程相关结构。 |
| 可以并行执行 | 页面主线程和工作者线程中的 JavaScript 可以同时推进。 |
| 可以共享部分内存 | 可以通过 `SharedArrayBuffer` 共享内存，并用 `Atomics` 做并发控制。 |

这些相似点解释了为什么工作者线程适合做计算密集任务。主线程不用等待计算完成，可以继续响应用户输入和页面渲染。

## 5. 🤔 工作者线程和传统线程有什么区别？

工作者线程不是传统意义上可以任意共享内存的线程。它有几个重要限制。

- 默认不共享完整内存，普通数据需要复制或转移。
- 不能直接访问主线程的 DOM。
- 不一定与页面运行在同一个进程中，具体取决于浏览器实现。
- 创建成本较高，不适合为大量细碎任务频繁创建。
- 运行环境比普通函数调用重，有独立的全局对象和事件循环。

这也是为什么工作者线程通常适合长期运行或处理较大任务。比如，为每个像素创建一个工作者线程来处理图片就不合适；更合理的做法是创建少量工作者线程，分块处理任务。

## 6. 🤔 工作者线程有哪些类型？

Web 工作者线程规范中主要定义了三类工作者线程。

| 类型           | 简介                                                 |
| -------------- | ---------------------------------------------------- |
| 专用工作者线程 | 最常见的 Worker，只能被创建它的页面或上下文使用。    |
| 共享工作者线程 | 可以被多个同源上下文共享，例如多个同源标签页。       |
| 服务工作者线程 | 主要用于拦截请求、缓存响应、支持离线访问和推送通知。 |

专用工作者线程和共享工作者线程更像后台脚本。服务工作者线程则更像网络代理，它的生命周期和使用方式都更加特殊。

## 7. 🤔 工作者线程里为什么没有 `window`？

页面脚本中的全局对象通常是 `window`。但工作者线程不运行在页面窗口里，因此没有 `window` 的概念。

工作者线程中的全局对象是 `WorkerGlobalScope` 的实例，并通过 `self` 暴露。

```js
// worker.js
console.log(self)
console.log(window) // ReferenceError
```

`self` 上暴露的是 `window` 能力的一个子集。常见属性包括：

| 属性              | 说明                           |
| ----------------- | ------------------------------ |
| `self`            | 指向当前工作者线程全局对象。   |
| `location`        | 当前工作者线程脚本的位置信息。 |
| `navigator`       | 工作者线程版本的导航器信息。   |
| `performance`     | 性能相关 API 的子集。          |
| `console`         | 输出调试信息。                 |
| `caches`          | 访问 `CacheStorage`。          |
| `indexedDB`       | 访问 IndexedDB。               |
| `isSecureContext` | 当前上下文是否安全。           |
| `origin`          | 当前全局对象的源。             |

常见方法包括：

- `setTimeout()`、`clearTimeout()`。
- `setInterval()`、`clearInterval()`。
- `fetch()`。
- `atob()`、`btoa()`。
- `createImageBitmap()`。

`WorkerGlobalScope` 还提供 `importScripts()`，用于在经典工作者线程中同步导入脚本。

## 8. 🤔 为什么每种工作者线程有不同的全局作用域？

不同类型的工作者线程需要不同的额外能力，因此它们的全局作用域都继承自 `WorkerGlobalScope`，再扩展自己的接口。

| 工作者线程     | 全局作用域                   |
| -------------- | ---------------------------- |
| 专用工作者线程 | `DedicatedWorkerGlobalScope` |
| 共享工作者线程 | `SharedWorkerGlobalScope`    |
| 服务工作者线程 | `ServiceWorkerGlobalScope`   |

例如，专用工作者线程可以直接使用 `postMessage()` 和创建者通信；共享工作者线程要通过 `connect` 事件拿到连接端口；服务工作者线程则会监听 `install`、`activate`、`fetch`、`push` 等事件。

抓住这一点，后面理解三类工作者线程会轻松很多：它们共享同一套后台执行模型，但暴露给你的入口和生命周期不同。
