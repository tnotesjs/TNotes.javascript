## 3. 动画与 Canvas 图形解决了什么问题？

现代 Web 应用经常需要动态图形，例如图表、游戏、图片编辑器、数据可视化、地图、视频特效和复杂动效。这类功能不能只依赖普通 DOM 节点完成，因为 DOM 更适合表达文档结构，而不是高频绘制像素。

本章讨论的核心就是两件事：

- 动画如何与浏览器重绘节奏保持一致。
- 图形如何通过 `<canvas>` 在页面中的一块绘图区动态生成。

早期 JavaScript 动画主要依赖 `setInterval()` 或 `setTimeout()`，但计时器只能把回调放入任务队列，无法保证回调正好发生在下一次重绘前。`requestAnimationFrame()` 的出现，让 JavaScript 动画可以接入浏览器自己的渲染调度。

`<canvas>` 则提供了一块可绘制区域。它本身只是元素，真正的绘制能力来自上下文对象。常见上下文包括：

| 上下文 | 主要用途 |
| --- | --- |
| `2d` | 绘制矩形、路径、文本、图片、渐变、图案和像素数据。 |
| `webgl` / `webgl2` | 基于 OpenGL ES 的 3D 图形绘制，也可以用于高性能 2D 图形。 |

## 4. `<canvas>` 在浏览器图形体系中处于什么位置？

`<canvas>` 是 HTML5 中引入的绘图元素。它会在页面中占据一块矩形区域，JavaScript 可以通过绘图上下文向这块区域写入图形。

它和普通 DOM 元素有一个很大的差异：普通 DOM 元素描述的是结构和样式，而 `<canvas>` 更像一块位图画板。你一旦把图形画上去，浏览器不会记住这个矩形、路径或文本的语义；如果下一帧要改变图形，通常需要重新绘制相关区域。

这也解释了为什么 Canvas 图形开发经常要考虑：

- 绘制频率，避免不必要的重绘。
- 坐标系统，明确每个图形落在画布上的位置。
- 状态管理，跟踪颜色、线条、变换和透明度等绘图状态。
- 跨源限制，避免画布被外部图片污染后无法导出像素数据。
- 硬件能力，尤其是 WebGL 中的缓冲区、纹理、着色器和 GPU 资源。

## 6. 本章小结要抓住哪些结论？

`requestAnimationFrame()` 是一个简单但很实用的工具。它把 JavaScript 动画安排在浏览器下一次重绘之前执行，因此比固定间隔计时器更适合视觉更新。它不只适合动画，也常用于滚动、拖拽、尺寸变化等高频 UI 更新的节流。

`<canvas>` 为 JavaScript 提供了动态创建图形的能力。它主要通过两类上下文工作：

- `2d` 上下文支持基本绘图，包括绘制矩形、路径、文本、图像，创建渐变和图案，读写像素，以及控制透明度和合成方式。
- WebGL 是浏览器对 OpenGL ES 的实现，提供更强大的 3D 图形能力，涉及定型数组、缓冲区、纹理、顶点着色器、片段着色器和 GLSL。

今天的主流浏览器都已经支持 `<canvas>` 和基本的 2D 绘图能力，较新的浏览器也普遍支持 WebGL。真正写图形程序时，难点通常不在于能不能画出来，而在于能不能稳定、准确、高性能地持续绘制。
