## 2. 🫧 评价

`<noscript>` 是个很小的元素，但它提醒你别把“脚本一定可用”当成天经地义。真正稳一点的页面，至少会给脚本失效时的用户留一扇门。

## 3. 🤔 `<noscript>` 解决的是什么问题？

`<noscript>` 出现的背景，是早期浏览器并不都支持 JavaScript。页面里如果直接放了脚本，不支持脚本的浏览器可能什么都做不了，甚至把脚本内容直接显示出来，页面体验会很差。

于是，HTML 提供了 `<noscript>` 作为兜底方案：当脚本不可用时，浏览器可以渲染一段替代内容，至少告诉用户当前页面发生了什么，或者给出下一步操作提示。

今天的浏览器几乎都支持 JavaScript，但这个元素并没有完全失去意义，因为“支持 JavaScript”和“当前脚本确实可用”不是一回事。

## 4. 🤔 浏览器什么时候会渲染 `<noscript>` 里的内容？

书里给出的条件很明确，满足任意一个即可：

- 浏览器不支持脚本
- 浏览器对脚本的支持被关闭

只要出现这两种情况之一，浏览器就会把 `<noscript>` 中的内容当作普通 HTML 渲染出来。

反过来，如果浏览器支持并启用了脚本，那么 `<noscript>` 中的内容通常不会显示。

这意味着你不能把 `<noscript>` 当成“平时也会露出来的一部分页面”，它本质上是一段条件性后备内容。

## 5. 🤔 `<noscript>` 里适合放什么内容？

`<noscript>` 可以包含几乎所有能出现在 `<body>` 里的常规 HTML 元素，唯独不应该再嵌套 `<script>`。它更适合承载的是“脚本不可用时用户最需要知道的信息”。

常见写法有三类：

- 一句明确提示，告诉用户当前页面依赖 JavaScript
- 一个最小可用替代方案，比如跳转链接或基础导航
- 一个降级版本说明，告诉用户某些功能暂时不可用

例如：

```html
<noscript>
  <p>这个页面需要启用 JavaScript 才能正常使用。</p>
  <p><a href="/help/enable-javascript">查看启用方法</a></p>
</noscript>
```

这种内容的目标不是“完整复刻脚本体验”，而是给用户一个明确、可执行的退路。

## 6. 🤔 今天还需要 `<noscript>` 吗？

从“浏览器是否支持 JavaScript”这个角度说，`<noscript>` 的历史任务确实已经弱了很多；但从“页面是否做好降级”这个角度说，它仍然值得保留在某些场景里。

比如下面这些页面就很适合考虑它：

- 核心流程强依赖脚本的页面
- 面向公共用户的表单、登录或支付入口
- 需要明确说明“脚本没开会导致页面不可用”的站点

当然，`<noscript>` 不是万能补丁。它不能替你完成真正的无脚本适配，也不能把复杂 Web 应用自动变成静态页面。它只是一个明确的兜底出口。

所以更准确的理解是：如果你的页面在脚本不可用时会明显失去基本可用性，那么至少应该考虑用 `<noscript>` 把情况说清楚。
