## 3. 为什么需要工作者线程？

前端开发中经常说 JavaScript 是单线程的。这句话并不描述浏览器内部的全部实现，但它很好地概括了页面脚本的常见行为：同一个页面里的 JavaScript 通常运行在一个主执行环境里，DOM、事件循环和用户交互都围绕这个环境展开。

如果一段脚本在主线程里执行大量计算，页面就会无法及时响应用户操作。

```js
// 主线程里执行过重计算，会拖慢页面响应
for (let index = 0; index < 1e9; index += 1) {
  // heavy work
}
```

工作者线程的价值就在这里：把不适合阻塞页面的任务放到独立环境中执行，让主线程继续负责渲染、事件处理和用户交互。

## 4. 工作者线程会改变 JavaScript 的单线程模型吗？

不会。工作者线程并不是让同一个页面脚本可以随意多线程访问 DOM，而是在页面主执行环境之外创建一个独立的 JavaScript 环境。

工作者线程有自己的：

- 全局对象。
- 事件循环。
- 内存环境。
- 消息队列。

它和主线程可以并行执行，但不能直接访问主线程中的 DOM。两个环境之间需要通过异步消息通信。

```js
// main.js
const worker = new Worker('./worker.js')

worker.postMessage({ type: 'calculate', value: 10 })
worker.onmessage = (event) => {
  console.log(event.data)
}
```

```js
// worker.js
self.onmessage = (event) => {
  self.postMessage(event.data.value * 2)
}
```

这种设计让浏览器既能保留 DOM 单线程操作的简单模型，又能把耗时任务交给后台环境。

## 5. 工作者线程有哪几类？

Web 工作者线程规范中主要有三类工作者线程。

| 类型 | 创建方式 | 典型用途 | 特点 |
| --- | --- | --- | --- |
| 专用工作者线程 | `new Worker()` | 复杂计算、数据处理、文件处理。 | 只服务于创建它的页面或上下文。 |
| 共享工作者线程 | `new SharedWorker()` | 多个同源页面共享连接或状态。 | 多个同源上下文可以连接到同一个线程。 |
| 服务工作者线程 | `navigator.serviceWorker.register()` | 请求拦截、离线缓存、推送通知。 | 更像网络代理，生命周期由浏览器管理。 |

这三类线程都运行在独立环境中，也都通过消息与外界通信；区别主要在谁能使用它、生命周期如何管理，以及它能拦截哪些浏览器行为。

## 6. 什么任务适合放进工作者线程？

工作者线程适合处理会让主线程卡顿的任务，例如：

- 大量数据排序、过滤、聚合。
- 图片、音视频、二进制数据处理。
- 加密、压缩、解压等计算密集任务。
- 复杂文本解析。
- 多页面共享的长连接管理。
- 离线缓存、请求代理和推送通知。

但工作者线程不是越多越好。创建工作者线程有成本，它需要独立的运行环境、事件循环和内存。通常应该让工作者线程处理较重、较长期或可批处理的工作，而不是为细碎任务频繁创建和销毁线程。

## 7. 本章小结要抓住哪些结论？

工作者线程可以运行异步 JavaScript，避免复杂计算和数据处理阻塞用户界面。它们拥有独立环境，不能直接操作 DOM，只能通过异步消息和外部通信。

专用工作者线程只属于创建它的上下文，适合把后台任务从页面主线程移走。共享工作者线程可以被同源的多个上下文连接，适合在多个页面之间共享线程资源或通信通道。

服务工作者线程的定位更特殊。它像浏览器中的网络代理，可以拦截请求、管理缓存、支持离线体验，也能配合 Push API 和 Notifications API 支持推送通知。它是渐进式 Web 应用的重要基础。
