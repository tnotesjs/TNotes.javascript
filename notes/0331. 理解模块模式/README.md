## 3. 🤔 模块模式的核心思想是什么？

模块模式的思想很简单：把逻辑拆成独立单元，每个单元内部封装实现，对外只暴露必要接口。

一个成熟的模块系统通常需要解决几件事：

- 如何识别一个模块。
- 如何声明模块依赖。
- 如何加载依赖。
- 如何决定入口模块。
- 如何处理循环依赖。
- 如何让工具分析模块关系。

也就是说，模块不是单纯的文件拆分。文件只是载体，真正重要的是模块边界、依赖关系和公共接口。

## 4. 🤔 什么是模块标识符？

模块标识符用于唯一引用一个模块。模块系统本质上可以看成一个以标识符为键的模块注册表。

不同系统中的标识符形式不同：

| 模块系统           | 常见标识符                                           |
| ------------------ | ---------------------------------------------------- |
| 浏览器原生 ES 模块 | 实际模块文件路径，例如 `./utils.js`。                |
| Node.js CommonJS   | 相对路径、绝对路径、包名，例如 `./utils`、`lodash`。 |
| AMD                | 字符串模块名，例如 `moduleA`。                       |

完善的模块系统必须避免标识冲突，并且能够把标识符解析为实际模块。

## 5. 🤔 模块依赖是什么？

模块依赖表示当前模块运行前必须准备好的其他模块。声明依赖，相当于模块和模块系统之间达成契约：当前模块告诉模块系统自己需要什么，模块系统负责加载和初始化这些依赖。

```js
import { formatDate } from './date.js'

console.log(formatDate(new Date()))
```

在这段代码里，当前模块依赖 `./date.js`。只有 `./date.js` 加载并完成求值之后，当前模块才能安全使用 `formatDate`。

依赖关系会形成依赖图。入口模块是图的起点，模块系统会从入口出发解析和加载所有依赖。

## 6. 🤔 模块加载和入口模块有什么关系？

入口模块是程序执行的起点。一个应用可以有一个入口，也可以有多个入口。入口模块可能依赖其他模块，其他模块又继续依赖更多模块，于是形成一张依赖图。

浏览器加载模块时，大致会经历：

1. 发现入口模块。
2. 下载入口模块代码。
3. 解析入口模块的依赖。
4. 下载并解析依赖模块。
5. 递归处理整个依赖图。
6. 按依赖关系完成模块求值。

如果没有模块系统，开发者可能需要手动排列脚本顺序。

```html
<script src="moduleE.js"></script>
<script src="moduleD.js"></script>
<script src="moduleC.js"></script>
<script src="moduleB.js"></script>
<script src="moduleA.js"></script>
```

这种方式脆弱又难维护。模块系统的价值之一，就是把这种顺序管理交给工具或运行时处理。

## 7. 🤔 异步依赖和动态依赖有什么区别？

异步依赖强调加载方式。模块可以请求加载另一个模块，并在加载完成后继续执行。

```js
load('moduleB').then((moduleB) => {
  moduleB.doStuff()
})
```

现代 ES 模块中的动态 `import()` 也是异步加载模块的一种方式。

```js
const module = await import('./moduleB.js')

module.doStuff()
```

动态依赖强调依赖是否在运行时才决定。

```js
if (condition) {
  require('./moduleA')
}
```

动态依赖灵活，但会让工具更难提前知道完整依赖图。静态依赖更容易被打包、预加载和优化。

## 8. 🤔 为什么静态分析对模块系统很重要？

静态分析是不执行代码，只根据代码结构推断依赖和行为。

对模块系统来说，静态分析能带来很多好处：

- 构建工具可以提前生成依赖图。
- 打包工具可以合并模块，减少请求数量。
- 编辑器可以做自动补全和跳转。
- 构建工具可以做摇树优化，删除未使用导出。

ES6 模块要求 `import` 和 `export` 出现在顶级，模块标识符也必须是静态字符串，这些限制就是为了让静态分析变得可靠。

## 9. 🤔 循环依赖为什么不可避免？

循环依赖指两个或多个模块互相依赖。例如：

```txt
moduleA -> moduleB -> moduleC -> moduleA
```

真实应用很难完全避免循环依赖，因此 CommonJS、AMD 和 ES6 模块都必须处理它。

循环依赖的问题在于加载顺序可能不直观。某个模块执行时，它依赖的模块可能还没有完成全部初始化。只要模块设计得足够干净，模块加载时没有过多副作用，循环依赖通常可以被模块系统处理。

经验上，要降低循环依赖风险，可以遵守几条原则：

- 把共享类型、常量或工具函数抽到更底层模块。
- 避免模块加载时立即执行复杂副作用。
- 用函数调用推迟依赖使用时机。
- 让模块公共接口稳定，不依赖初始化过程中的临时状态。
