## 3. 这一章到底在解决什么问题？

这一章的核心问题是：JavaScript 里对象是怎么组织的、怎样批量创建类似对象、这些对象如何共享行为，以及类到底在语义上做了什么。

如果把这一章拆开看，会得到四条连续主线：

- 对象属性并不只是简单键值对，它还有可写、可枚举、可配置、访问器等特性。
- 当你需要创建很多结构相似的对象时，必须考虑复用和标识问题。
- 继承在 JavaScript 里主要围绕原型链展开，而不是传统类系统先验存在。
- `class` 把前面这些底层机制包装成了更清晰的语法，但底层仍然是原型和构造函数。

## 4. 这一章最值得抓住的四条主线是什么？

| 主线     | 关注点                                                         |
| -------- | -------------------------------------------------------------- |
| 对象模型 | 属性描述符、访问器、对象合并、解构和增强语法                   |
| 创建模式 | 工厂模式、构造函数模式、原型模式及其组合                       |
| 继承模式 | 原型链、盗用构造函数、组合继承、寄生式组合继承                 |
| 类语法   | `class`、`constructor`、实例成员、静态成员、`extends`、`super` |

很多人学习这一章时会把“对象”和“类”分成两套逻辑，但 JavaScript 的真实情况恰恰相反：类是后来的语法糖，对象和原型机制才是底层地基。

## 5. 为什么说 ES6 类不是完全新的对象系统？

因为 `class` 并没有创造出一套脱离原型的新运行时。它主要做的是：

- 让定义构造函数、原型方法、静态方法和继承关系的语法更清晰。
- 把一些早期模式里容易写乱的细节收束到固定语法里。
- 在调用约束上更严格，比如类构造函数必须通过 `new` 调用。

所以理解类时，最好的路径不是忘掉构造函数和原型，而是先知道它们，再把类看成更现代的表达层。

## 6. 这一章应该怎样收束成稳定结论？

可以把这一章收束成下面几条：

- JavaScript 对象是动态的，属性行为可以通过描述符精细控制。
- 批量创建对象时，核心矛盾是“实例私有数据”和“共享方法复用”如何平衡。
- JavaScript 的继承本质上建立在原型链之上。
- ES6 类让对象创建和继承更易写、更易读，但底层机制没有脱离原型体系。

把这些点串起来，后面再看代理、函数、模块和框架里的面向对象写法时，你会更容易判断哪些是语言本质，哪些只是语法层包装。
