## 2. 🫧 评价

- 事件性能优化最重要的动作通常不是“让回调更快”，而是少绑一些回调、把监听放在稳定节点上，并在不需要时干净地移除。

## 3. 🤔 为什么事件处理程序会影响内存和性能？

事件处理程序是函数对象，会占用内存。绑定事件还常常伴随 DOM 查询、闭包引用和初始化逻辑。

如果页面上给大量节点分别绑定事件，问题会变多：

- 初始化更慢。
- 内存占用更多。
- 动态新增节点还要重复绑定。
- 节点删除时更容易遗留引用。

因此，事件优化的第一步通常是减少监听器数量。

## 4. 🤔 什么是事件委托？

事件委托利用事件冒泡，把多个子元素的同类事件交给共同祖先处理。

例如，不给每个 `li` 单独绑定点击事件，而是给 `ul` 绑定一次。

```js
list.addEventListener('click', (event) => {
  const item = event.target.closest('li')

  if (!item || !list.contains(item)) {
    return
  }

  console.log(item.dataset.id)
})
```

这样无论列表里有多少项，都只需要一个监听器。

## 5. 🤔 事件委托有哪些优点？

事件委托的优点很直接：

- 减少监听器数量。
- 降低内存占用。
- 减少重复 DOM 查询和绑定代码。
- 动态新增子元素通常不需要重新绑定。
- 监听器可以放在更稳定的祖先节点上。

事件委托特别适合列表、菜单、表格、树形结构和工具栏这类重复元素。

```js
toolbar.addEventListener('click', (event) => {
  const button = event.target.closest('button[data-command]')

  if (button) {
    runCommand(button.dataset.command)
  }
})
```

## 6. 🤔 哪些事件适合委托？

适合委托的事件通常有两个特点：会冒泡，并且多个子元素有相似逻辑。

常见适合委托的事件包括：

- `click`
- `mousedown`
- `mouseup`
- `keydown`
- `keyup`
- `input`
- `change`

有些事件虽然可处理，但需要额外小心。例如 `mouseover` 和 `mouseout` 会在子元素之间移动时频繁触发，常常需要结合 `relatedTarget` 判断是否真的离开了容器。

`focus` 和 `blur` 不冒泡，如果要委托焦点变化，通常使用 `focusin` 和 `focusout`。

## 7. 🤔 什么时候不适合事件委托？

事件委托不是万能的。

下面这些场景可能不适合：

- 子元素逻辑差异很大，委托处理函数会变成复杂分支。
- 事件不冒泡，且没有合适的冒泡替代事件。
- 需要非常精确地控制某个元素的生命周期。
- 目标元素可能在封装边界内，事件无法自然冒泡出来。

如果委托函数里全是复杂判断，反而不如局部绑定清晰。

## 8. 🤔 为什么要删除事件处理程序？

当事件处理程序不再需要时，应该移除它。

```js
function handleClick(event) {
  console.log(event.target)
}

button.addEventListener('click', handleClick)

// 不再需要时
button.removeEventListener('click', handleClick)
```

如果节点被 `removeChild()`、`replaceChild()` 或 `innerHTML` 删除，而事件处理程序仍然持有外部引用，旧浏览器和复杂页面中可能出现内存无法释放的问题。

现代框架通常会在组件卸载时做清理，但手写 DOM 代码仍要自己负责。

## 9. 🤔 移除 DOM 前应该怎么处理事件？

如果你直接绑定了节点事件，移除节点前最好先移除处理程序。

```js
button.removeEventListener('click', handleClick)
button.remove()
```

如果使用事件委托，监听器放在稳定祖先上，子节点删除时通常不需要逐个解绑。

这也是事件委托的一个隐藏优势：它把监听器放在更稳定的位置，减少动态 DOM 的清理成本。

## 10. 🤔 页面卸载时需要清理事件吗？

传统建议是：页面加载阶段注册的事件，在卸载阶段尽量恢复或清理。

但现代浏览器里，`unload` 事件可能影响往返缓存，很多场景不建议依赖它做复杂逻辑。

更稳妥的思路是：

- 组件或模块销毁时立即清理自己的监听器。
- 页面生命周期相关逻辑优先考虑 `pagehide`、`visibilitychange` 等更现代事件。
- 不把重要业务逻辑放在卸载事件里。

事件清理的核心不是追求某个固定时机，而是让监听器的生命周期和它依赖的 DOM、组件、数据保持一致。

## 11. 🤔 事件性能有哪些实用建议？

可以按这几条做：

- 重复元素优先考虑事件委托。
- 高频事件中避免大量 DOM 读写。
- 需要移除的处理程序保留函数引用。
- 动态删除 DOM 前清理直接绑定的监听器。
- 不在循环里给大量节点随意绑定匿名函数。
- 滚动、鼠标移动、窗口尺寸变化等高频事件需要节流、合并或延迟处理。

事件性能优化的关键不是把每个回调写得极端复杂，而是让事件绑定结构本身足够克制。
