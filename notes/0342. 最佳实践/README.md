## 3. 为什么需要 JavaScript 最佳实践？

早期网页中的 JavaScript 常用于表单验证、小动画和简单交互。如今，JavaScript 已经承担复杂 Web 应用的核心逻辑，代码量、团队规模和运行环境都比过去复杂得多。

这种变化让 JavaScript 开发不再只是把效果写出来，而是要考虑代码能否长期维护、运行是否足够快、交付到线上是否可靠。

本章把最佳实践分为三个方向：

| 方向     | 关注点                                           |
| -------- | ------------------------------------------------ |
| 可维护性 | 代码是否容易理解、修改、扩展和调试。             |
| 性能     | 代码执行、DOM 交互和数据处理是否高效。           |
| 部署     | 源代码如何经过验证、构建、压缩和传输优化后发布。 |

简单来说，最佳实践要解决的是从能运行到能长期交付的问题。

## 4. 可维护性关注什么？

可维护代码应该具备几个特征：

- 容易理解：别人不用找原作者，也能看懂代码意图。
- 符合常识：代码结构和命名都顺理成章。
- 容易适配：数据或需求变化时，不需要整块重写。
- 容易扩展：架构能支持未来功能增长。
- 容易调试：出错时能提供足够线索定位问题。

JavaScript 的灵活性很高，这既是优势，也是维护风险。团队需要用编码规范、命名规则、注释策略、职责分离和编码惯例来约束代码。

特别是在 Web 应用中，HTML、CSS 和 JavaScript 应该各司其职：

| 层         | 职责             |
| ---------- | ---------------- |
| HTML       | 表达内容和结构。 |
| CSS        | 表达显示和外观。 |
| JavaScript | 表达行为和交互。 |

如果三者混在一起，排查问题时就很难判断错误来自结构、样式还是行为。

## 5. 性能优化关注什么？

随着页面中的 JavaScript 代码量增长，代码执行时间会直接影响网页响应速度。现代 JavaScript 引擎已经很快，但仍然可能因为不合适的写法、算法或 DOM 操作拖慢页面。

本章强调的性能方向包括：

- 减少全局查找和不必要的作用域链查找。
- 避免 `with`、`eval()`、字符串形式的 `setTimeout()` 等会拖慢解析的写法。
- 避免重复属性查找，把频繁访问的值保存到局部变量。
- 优化循环，避免把可移出的计算放在循环体里。
- 在适合时减少语句数量。
- 限制 DOM 实时更新次数。
- 使用事件委托减少事件处理程序数量。
- 谨慎访问实时集合，例如 `HTMLCollection`。

不过性能优化也要服务于实际问题。为了微小收益牺牲可读性，往往得不偿失。真正应该优先优化的是热点路径、大数据处理和高频 DOM 更新。

## 6. 部署阶段为什么也是最佳实践的一部分？

开发时的代码通常强调可读性和可维护性；部署时的代码则更强调体积、兼容性、加载速度和稳定性。这两者不应该混为一谈。

典型部署流程会包含：

- 拆分源码，便于维护。
- 构建时打包为较少的输出文件，便于浏览器加载。
- 运行代码检查和测试，提前发现问题。
- 摇树优化，移除未使用代码。
- 最小化和压缩 JavaScript。
- 必要时转译为目标浏览器支持的语法。
- 启用 HTTP 压缩，减少网络传输负载。

也就是说，部署不是简单地把开发文件上传到服务器，而是把源码处理成适合浏览器和线上环境消费的产物。

## 7. 本章小结要抓住哪些结论？

随着 JavaScript 开发日益成熟，前端开发已经和其他软件开发一样，需要认真对待可维护性、性能优化和部署流程。

为了保证可维护性，可以借鉴其他语言的编码惯例，比如统一缩进、补充必要注释、使用清晰命名。但 JavaScript 也有自己的特殊要求，例如尽量减少全局变量、不要修改不属于自己的对象，并保持 HTML、CSS 和 JavaScript 的职责分离。

随着 Web 应用中的 JavaScript 代码量增加，性能也越来越重要。执行 JavaScript 所需时间会直接影响用户体验，DOM 交互也往往是性能热点。优化时应优先减少不必要的作用域查找、属性查找、循环成本和 DOM 更新次数。

上线部署是 Web 应用开发的最后一步。合理的构建流程可以自动完成代码检查、打包、转译、压缩等任务。JavaScript 文件应在部署前最小化，并通过 HTTP 压缩降低传输负载，从而提升页面整体性能。
