## 3. 为什么还要了解旧版 IE 错误？

旧版 IE 曾经是最难调试 JavaScript 的浏览器之一。它的错误消息经常很短、很模糊，缺少调用栈和上下文。

今天大多数项目已经不再支持旧版 IE，但这些历史问题仍然能提醒你：浏览器报错不一定直接指向根因。看到错误消息时，需要结合资源内容、DOM 结构、字符编码、请求参数和浏览器实现一起判断。

## 4. 什么是无效字符错误？

无效字符错误通常表示 JavaScript 文件里出现了语言语法不接受的字符。

常见例子是从富文本软件复制代码时，把普通减号 `-` 复制成了外观相似但 Unicode 编码不同的字符。这个字符看起来像减号，但 JavaScript 语法并不把它当作减号。

```js
// 看起来像减法，但中间字符可能不是 ASCII 减号
const result = total - discount
```

这类问题在旧版 IE 中可能显示为无效字符，其他浏览器可能显示非法字符、语法错误或引用错误。

解决方式包括：

- 不从富文本软件复制代码。
- 使用纯文本编辑器或代码编辑器。
- 确保文件编码一致。
- 让代码检查工具和格式化工具参与构建流程。

## 5. 未找到成员错误是怎么来的？

旧版 IE 中很多 DOM 对象由 COM 对象实现，不完全等同于原生 JavaScript 对象。在对象生命周期和垃圾回收交错时，可能出现未找到成员错误。

一个典型场景是把旧版 IE 的事件对象保存到异步闭包里。事件处理结束后，事件对象已经被销毁，后续再访问它的成员就可能报错。

```js
document.onclick = function () {
  var event = window.event

  setTimeout(function () {
    event.returnValue = false
  }, 1000)
}
```

更稳妥的做法是只保存真正需要的普通值，而不是保存整个宿主对象。

```js
document.onclick = function () {
  var target = window.event.srcElement

  setTimeout(function () {
    if (target) {
      target.className += ' clicked'
    }
  }, 1000)
}
```

这个问题的现代意义是：不要假设宿主对象和普通对象完全一样。涉及异步、事件对象和 DOM 对象时，最好提取必要数据，减少对临时对象生命周期的依赖。

## 6. 未知运行时错误通常和什么有关？

旧版 IE 中，使用 `innerHTML` 或 `outerHTML` 插入不符合 HTML 结构规则的内容时，可能出现未知运行时错误。

例如，把块级元素插入到不允许包含块级元素的位置，或在表格相关元素上直接写入不合适的 HTML，都可能触发这类问题。

```js
// 在旧版 IE 中，这类不合法结构可能导致未知运行时错误
paragraph.innerHTML = '<div>Hi</div>'
```

其他浏览器可能会自动纠正 DOM 结构，但旧版 IE 更严格，也更容易以模糊错误消息失败。

解决思路是：

- 插入 HTML 前确认目标元素允许对应子元素。
- 表格结构优先使用 DOM 方法构建。
- 对复杂片段先放入合适容器，再移动节点。
- 不依赖浏览器自动修复非法结构。

## 7. 为什么外部脚本会导致语法错误？

如果页面通过脚本资源引用了非 JavaScript 内容，旧版 IE 可能只报告语法错误，而且位置常常是第一行第一个字符。

典型原因包括：

- `src` 指向了 HTML 页面。
- 服务器错误页返回了 HTML。
- 动态生成脚本时后端报错，把错误页面输出成脚本内容。
- 资源路径错误，返回了登录页或 404 页面。

排查这类问题时，不要只看报错行号，还要打开 Network 面板或直接访问脚本 URL，确认响应内容确实是 JavaScript。

## 8. 系统找不到指定资源和 URL 长度有什么关系？

旧版 IE 对 URL 长度限制比较严格。超过限制的 URL 可能导致系统找不到指定资源这类模糊错误。

这种问题常出现在把大量数据拼到查询字符串里时。

```js
var xhr = new XMLHttpRequest()

xhr.open('GET', veryLongUrl, true)
xhr.send(null)
```

解决方式包括：

- 缩短查询参数名和值。
- 去掉不必要的数据。
- 把大量数据改为请求体提交。
- 对复杂数据使用 `POST`。

今天虽然很少因为 IE 限制遇到这个错误，但 URL 过长依然不是好设计。请求语义、缓存、安全日志和服务器限制都可能受到影响。

## 9. 这些历史错误给现代开发什么启发？

第一，错误消息可能只是症状，不一定是根因。

第二，资源内容比资源扩展名更重要。脚本 URL 返回 HTML，本质上就不是脚本。

第三，DOM 插入要遵守结构规则，不要依赖浏览器替你修复非法 HTML。

第四，输入和传输边界要受控。字符编码、URL 长度、请求数据格式都可能成为错误来源。

第五，工具链很重要。现代编辑器、格式化器、linter、类型系统、构建检查和浏览器开发者工具，解决的正是过去那些难以定位的问题。
