## 3. 这一章解决了什么问题？

JavaScript 是单线程执行模型，但很多任务并不会立即完成，比如计时器、网络请求、文件读取、用户交互和后台任务。如果让主线程一直等待这些任务结束，页面和程序都会变得卡顿。

异步编程要解决的问题就是：当前代码先继续执行，等未来某个结果可用时，再用合适的方式接回来。

早期 JavaScript 主要依赖回调函数表达异步流程。回调能工作，但在多步骤依赖、错误处理和任务组合方面很快会变得混乱。

## 4. 期约和异步函数分别扮演什么角色？

`Promise` 用一个对象表示尚未完成的异步结果。它有明确的状态机，可以进入兑现或拒绝状态，并通过 `then()`、`catch()`、`finally()` 接收后续处理。

`async`/`await` 则是在函数语法层面对期约进行包装。它让你可以用更接近同步代码的结构来组织异步逻辑，但执行过程仍然是异步的，不会阻塞主线程。

可以这样理解：

| 能力            | 重点                                 |
| --------------- | ------------------------------------ |
| 回调            | 异步完成后调用某个函数               |
| `Promise`       | 用对象表示异步结果，并支持连锁和合成 |
| `async`/`await` | 用函数暂停和恢复的语法组织期约流程   |

## 5. 本章学习后应该留下哪些结论？

本章可以收束为几条关键结论：

- 异步操作不是另一条 JavaScript 主线程，它是通过运行时调度，在将来某个时机恢复执行。
- 回调是异步编程的早期基础，但嵌套回调难以扩展。
- `Promise` 是对尚未可用结果的抽象，状态一旦落定就不可逆。
- 期约处理程序总是异步排期执行，即使期约已经落定也不会同步重入当前调用栈。
- `then()`、`catch()`、`finally()` 都会返回新期约，因此可以连锁。
- `Promise.all()` 适合等待一组任务全部完成，`Promise.race()` 适合关注最先落定的任务。
- `async` 函数始终返回期约，`await` 会暂停当前异步函数并在之后恢复执行。
- `await` 不会阻塞主线程，但连续 `await` 可能把本可并行的任务写成串行。

简单来说，期约提供异步结果模型，异步函数提供更好读的异步控制流。两者合在一起，是现代 JavaScript 写异步代码的主力方式。
