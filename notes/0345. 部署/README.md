## 2. 评价

- 部署不是把源码扔到服务器上就结束；它是把面向开发者的代码，转换成面向浏览器、网络和真实用户的交付产物。

## 3. 为什么部署前需要构建流程？

开发阶段的代码通常追求可读、可维护、易调试；线上环境需要的是体积小、加载快、兼容目标浏览器、稳定可靠的产物。

如果直接把源码发布到线上，会遇到几个问题：

- 注释和清晰命名方便开发者，但增加线上文件体积。
- 分散的文件组织方便维护，但可能增加浏览器请求数量。
- 源码可能包含调试代码、未使用代码或潜在语法问题。
- 现代语法不一定被目标浏览器全部支持。

构建流程的作用，就是在发布前自动完成检查、转换和优化，把源码变成适合线上交付的资源。

## 4. 源码文件应该如何组织？

从维护角度看，不应该把所有 JavaScript 都塞进一个文件。更好的方式是按模块、类型或功能拆分文件，让每个文件只承担有限职责。

```txt
src/
  api/
    user.js
  components/
    user-list.js
  utils/
    date.js
  main.js
```

这样做有几个好处：

- 单个文件更短，更容易理解。
- 修改影响范围更小。
- 多人协作时更少出现合并冲突。
- 依赖关系更清楚。

但从部署角度看，文件越多不一定越好。浏览器请求数量、脚本加载顺序和阻塞行为都会影响性能。因此，源码可以分散维护，构建产物则通常会打包成一个或少数几个文件。

## 5. 任务运行器和构建工具负责什么？

当项目变大后，发布前需要做的事情会越来越多，例如：

- 代码检查。
- 单元测试。
- 类型检查。
- 模块打包。
- 语法转译。
- 资源压缩。
- 本地开发服务器。
- 上传部署。

任务运行器和构建工具可以把这些步骤自动化。原书提到 Grunt 和 Gulp；现代项目中也常见 Vite、Webpack、Rollup、esbuild、Rspack 等工具。

无论工具如何变化，核心目标都一样：让复杂、重复、容易出错的发布步骤变成稳定、可重复执行的流程。

```json
{
  "scripts": {
    "lint": "eslint src",
    "test": "vitest run",
    "build": "vite build"
  }
}
```

## 6. 什么是摇树优化？

摇树优化用于移除没有被使用的代码。它依赖静态模块结构，让构建工具可以判断哪些导出从未被引用。

```js
// utils.js
export const used = 'used'
export const unused = 'unused'
```

```js
// main.js
import { used } from './utils.js'

console.log(used)
```

如果构建工具能确认 `unused` 没有被使用，就可以把它从最终产物中排除。

摇树优化的效果取决于几个因素：

- 使用静态 `import` 和 `export`。
- 模块没有难以分析的副作用。
- 构建工具能理解依赖关系。
- 包本身提供适合摇树优化的模块产物。

摇树优化不是魔法。动态导入、运行时属性访问和带副作用模块都可能限制优化效果。

## 7. 为什么需要模块打包器？

以模块形式写代码，不代表一定要以大量模块文件交付给浏览器。模块打包器会分析依赖关系，把多个模块组合成适合发布的产物。

打包器通常负责：

- 从入口文件开始分析依赖图。
- 合并 JavaScript 模块。
- 处理 CSS、图片、字体等资源引用。
- 分割代码块，支持按需加载。
- 输出带哈希的静态资源文件。

常见打包工具包括 Webpack、Rollup、Browserify，以及现代开发中常见的 Vite、esbuild、Rspack 等。

打包的目标不是一定生成单个文件，而是根据应用结构输出更适合缓存、并行加载和按需加载的少量资源。

## 8. 部署前为什么要做代码验证？

只靠在浏览器里手动刷新页面，很难发现所有问题。没有执行到的代码路径不会报错，手动验证也不容易自动化。

代码检查工具可以在构建前发现潜在问题，例如：

- 使用未声明变量。
- 错误使用等号。
- 不合适的换行或语法。
- 遗漏 `switch` 分支中的 `break`。
- 重复声明变量。
- 使用 `eval()` 或 `with`。
- 无法执行到的代码。

现代项目通常会使用 ESLint、TypeScript、测试框架和 CI 流程组合验证。

```bash
pnpm lint
pnpm test
pnpm build
```

验证越早自动化，线上问题就越少。构建流程中加入验证步骤，可以在发布前拦住明显错误。

## 9. 什么是代码压缩和最小化？

JavaScript 以源码形式交给浏览器。源码中有很多对浏览器执行没帮助、但对开发者阅读有帮助的内容，例如空格、换行、注释、长变量名。

最小化会在保持语法正确和行为一致的前提下缩小代码体积，常见操作包括：

- 删除空格和换行。
- 删除注释。
- 缩短局部变量名和函数名。
- 简化表达式。

开发代码：

```js
function calculateTotalPrice(items) {
  return items.reduce((total, item) => total + item.price, 0)
}
```

最小化后可能变成：

```js
function calculateTotalPrice(t) {
  return t.reduce((t, r) => t + r.price, 0)
}
```

原书中有时把 compression 和 minification 都称作压缩。严格来说，最小化后的代码仍然是合法 JavaScript；HTTP 压缩后的内容则需要浏览器解压后才能还原。

## 10. JavaScript 编译和转译有什么区别？

JavaScript 编译在 Web 构建语境中，通常指把源码转换为更小或更适合执行的等价代码。构建工具可能会：

- 删除未使用代码。
- 内联常量。
- 简化控制流。
- 改写表达式。
- 合并或重排模块结构。

转译则通常指把较新的 JavaScript 语法转换为目标环境支持的旧语法。

```js
// 源码
const doubled = values.map((value) => value * 2)
```

可能转为：

```js
var doubled = values.map(function (value) {
  return value * 2
})
```

转译让开发者可以使用较新的语言特性，同时兼容目标浏览器。实际项目中常通过 Babel、TypeScript、SWC、esbuild 等工具完成。

是否需要转译，取决于目标浏览器范围。只面向现代浏览器的项目，可以少转译甚至不转译；需要支持旧环境的项目，则要更保守。

## 11. HTTP 压缩解决什么问题？

最小化减少的是浏览器需要解析的代码大小。HTTP 压缩减少的是服务器实际传给浏览器的字节数，也就是传输负载。

浏览器会通过请求头说明自己支持哪些压缩格式。

```txt
Accept-Encoding: gzip, br
```

服务器返回压缩后的资源时，会通过响应头说明压缩方式。

```txt
Content-Encoding: br
```

浏览器收到后会自动解压，再交给 JavaScript 引擎解析执行。

常见压缩格式包括 `gzip`、`br` 等。JavaScript 是文本格式，压缩率通常很高。虽然服务器压缩和浏览器解压也有成本，但多数情况下，减少网络传输字节带来的收益更明显。

## 12. 一个合理的发布流程通常长什么样？

一个典型的 JavaScript 发布流程可以概括为：

1. 编写模块化源码。
2. 运行格式检查、代码检查和测试。
3. 根据目标环境转译语法。
4. 打包模块和资源。
5. 摇树优化，移除未使用代码。
6. 最小化 JavaScript、CSS 和 HTML。
7. 给静态资源加内容哈希，方便长期缓存。
8. 在服务器或 CDN 上启用 HTTP 压缩。
9. 发布后监控错误和性能指标。

具体项目可能会增删步骤，但方向一致：源码服务于开发效率，构建产物服务于运行效率和交付稳定性。
