## 2. 🫧 评价

- 作用域是 JavaScript 里最值得你反复咀嚼的基础概念之一。很多“为什么这里能访问、那里不能访问”的问题，最后都不是语法题，而是执行上下文和标识符查找路径没想清楚。

## 3. 🤔 执行上下文到底是什么？

执行上下文可以理解为“当前这段代码运行时所处的环境”。这个环境决定了：

- 当前能访问哪些变量和函数。
- 这些变量和函数优先从哪里查找。
- 代码执行完之后，这个环境什么时候退出。

最外层的是全局上下文。进入函数时，会创建函数上下文；进入块级作用域时，`let` 和 `const` 也会形成更细的词法边界。

```js
const color = 'blue'

function changeColor() {
  const anotherColor = 'red'
  return anotherColor
}
```

这里全局上下文里有 `color` 和 `changeColor`，函数上下文里有 `anotherColor`。函数内部可以向上访问全局作用域，但全局代码不能反过来直接访问函数内部变量。

代码执行时，上下文会按调用顺序入栈、出栈。谁最后进入，谁最先退出。

## 4. 🤔 作用域链是怎样决定变量查找顺序的？

当代码里出现一个标识符时，JavaScript 引擎会从当前上下文开始查找。如果当前层没有，再沿着外层作用域逐级向上找，这条查找路径就是作用域链。

```js
const color = 'blue'

function getColor() {
  return color
}

console.log(getColor()) // 'blue'
```

在 `getColor()` 中查找 `color` 时，函数自己的局部作用域里没有，于是继续向上找到全局作用域中的 `color`。

一旦当前层找到了同名变量，搜索就会停止：

```js
const color = 'blue'

function getColor() {
  const color = 'red'
  return color
}

console.log(getColor()) // 'red'
```

这就是变量遮蔽。离你最近的那个同名标识符，会屏蔽外层的同名变量。

## 5. 🤔 哪些场景会临时增强作用域链？

有些语句会在当前查找链前面临时插入一层可访问对象。

书里重点提了两类：

- `catch` 块
- `with` 语句

其中 `with` 会把指定对象临时推到查找链前面：

```js
function buildUrl() {
  const qs = '?debug=true'

  with (location) {
    return href + qs
  }
}
```

在这个例子里，`href` 会优先被解释成 `location.href`。

不过这类能力在现代代码里几乎不该主动使用，尤其 `with` 会让变量解析变得模糊，而且严格模式直接禁止它。你更应该把它当作“理解旧代码时要知道”的知识，而不是推荐写法。

## 6. 🤔 `var`、`let`、`const` 的作用域差异到底是什么？

三者最大的差异不是“能不能重新赋值”这么简单，而是它们绑定在哪一层作用域，以及声明前后能不能安全访问。

| 关键字  | 作用域     | 重复声明 | 声明前访问             |
| ------- | ---------- | -------- | ---------------------- |
| `var`   | 函数作用域 | 允许     | 结果通常是 `undefined` |
| `let`   | 块级作用域 | 不允许   | 进入暂时性死区         |
| `const` | 块级作用域 | 不允许   | 进入暂时性死区         |

`var` 的一个典型特点是声明提升：

```js
console.log(name) // undefined
var name = 'Jake'
```

这段代码不会报 `ReferenceError`，因为 `var` 声明会提升到作用域顶部，只是赋值还留在原位置。

而 `let` 和 `const` 不一样。它们也会进入当前词法环境，但在声明语句执行前属于暂时性死区，不能访问：

```js
console.log(age) // ReferenceError
let age = 18
```

`const` 还多一个要求：声明时必须初始化，而且后续不能重新绑定到新值。

## 7. 🤔 标识符查找为什么会直接影响代码行为？

因为你写下一个变量名时，JavaScript 并不是“看起来像哪个就用哪个”，而是严格按查找链解析。

```js
const color = 'blue'

function getColor() {
  const color = 'red'

  {
    const color = 'green'
    return color
  }
}

console.log(getColor()) // 'green'
```

这段代码最终返回 `'green'`，不是因为它“更晚声明”，而是因为返回语句所在位置最近能找到的 `color` 就是块级作用域里的那个。

理解这一点后，你会更容易解释下面这些现象：

- 为什么循环里用 `let` 和用 `var` 行为不同。
- 为什么闭包拿到的是某一层作用域中的变量，而不是变量当时的字面值快照。
- 为什么同名局部变量会屏蔽外层变量。

顺手再记一个实践结论：局部变量通常比全局变量更容易读，也更容易让引擎优化，因为查找路径更短、更明确。
