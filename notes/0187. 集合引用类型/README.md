## 2. 🫧 评价

## 3. 🤔 什么叫集合引用类型？

集合引用类型指的是那些专门用于组织、存放、访问和遍历多份数据的对象类型。

这一章里最核心的集合有三类：

- 以属性组织数据的 `Object`
- 以顺序组织数据的 `Array` 和定型数组
- 以键值关系或唯一值关系组织数据的 `Map`、`WeakMap`、`Set`、`WeakSet`

它们都能“装数据”，但组织方式、访问方式和生命周期管理完全不同。

## 4. 🤔 这一章的集合大致可以怎么分组？

可以按使用目标把它们分成四组：

| 类型               | 更适合处理什么                                   |
| ------------------ | ------------------------------------------------ |
| `Object`           | 结构清晰、字段固定或半固定的名值数据             |
| `Array`            | 有顺序的数据列表                                 |
| 定型数组           | 二进制、数值密集、性能敏感的数据                 |
| `Map` / `Set` 家族 | 键值映射、去重集合，以及依赖对象生命周期的弱集合 |

这里最容易混淆的是：

- `Object` 和 `Map` 都像键值容器，但键的能力不同。
- `Array` 和 `Set` 都能装多个值，但一个强调顺序，一个强调唯一性。
- `Map` / `Set` 和它们的弱版本 API 看起来相似，但弱集合不能枚举内容。

## 5. 🤔 学这一章时最关键的选择题是什么？

这一章真正的主线可以压缩成三道选择题。

第一道，键值存储要不要支持任意类型键？如果要，用 `Map`；如果只是少量固定字段，用 `Object` 往往更直接。

第二道，你要的是“可重复的有序序列”，还是“自动去重的唯一值集合”？前者是 `Array`，后者是 `Set`。

第三道，这份关联数据是否应该随着对象一起被垃圾回收？如果答案是“是”，那就要考虑 `WeakMap` 或 `WeakSet`。

很多集合 API 看起来像功能差不多，实际上是在回答不同的数据组织问题。

## 6. 🤔 这一章应该怎样收束成稳定结论？

可以把本章收束成下面几条：

- `Object` 是最基础的键值对象，但不等于最好的映射结构。
- `Array` 是最常用的顺序容器，方法多、表达力强。
- 定型数组服务的是二进制和性能场景，不是普通业务列表。
- `Map` / `Set` 解决的是更明确的键值映射和唯一值集合问题。
- `WeakMap` / `WeakSet` 则把垃圾回收也纳入了数据结构设计的一部分。
- 迭代器和扩展操作让这些集合之间可以更方便地互转、复制和遍历。

把这些结论连起来，后面你在设计数据结构时就不会只盯着“能不能装进去”，而会开始考虑“该用什么集合来装”。
