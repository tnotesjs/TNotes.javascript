## 1. 本节内容

- 行内脚本与外部脚本的两种形态
- 两者在维护和加载上的差异
- 为什么通常更推荐外部文件
- `src` 与行内代码的关系
- 多脚本拆分与缓存收益

## 2. 评价

这一节不是在讲“行内一定错、外链一定对”，而是在帮你建立取舍标准。把脚本放哪儿，本质上是在平衡维护成本、缓存收益和页面加载行为。

在 TNotes.javascript 中，会优先采用行内代码的方式来展示示例，因为规模很小，且利于教学演示；但对于真正需要维护的业务代码，还是应该把它放在独立的外部文件里，这样更符合实际项目的组织方式，也更利于后续维护和重构。

## 3. 行内代码和外部文件分别是什么？

把 JavaScript 放进页面，最直接有两种方式：

- 行内脚本：把代码直接写在 script 元素的起止标签之间
- 外部脚本：通过带 `src` 属性的 script 元素引入单独的脚本文件

它们都能运行 JavaScript，但管理方式完全不同。

- 行内脚本更像“代码跟着页面走”，适合非常短、非常局部的逻辑
- 外部脚本更像“页面去引用一段独立资源”，适合真正需要维护的业务代码

行内代码和外部文件无法同时存在：一旦你使用了 `src`，就不应该再在这个 script 元素内部写正式的脚本主体。浏览器会优先处理外部文件，而忽略中间的行内代码。

## 4. 为什么通常更推荐外部文件？

- 第一，外部文件更容易维护。JavaScript 如果散落在很多 HTML 页面里，排查和修改都会越来越痛苦；而把脚本收敛到独立文件中，你就可以单独编辑、复用和组织它们。
- 第二，外部文件可以被缓存。如果多个页面都引用同一个脚本文件，浏览器通常只需要下载一次。后续页面再用到它，成本就会低很多。
- 第三，外部文件让 JavaScript 和 HTML 各自独立演进。页面结构调整不影响脚本，脚本重构也不需要改动 HTML。外部文件与 HTML 解耦，迁移和重构时更灵活。

综上，“尽量用外部文件”并不是道德口号，而是基于维护性和缓存机制得出的工程判断。

## 5. 行内代码什么时候仍然有存在感？

推荐外部文件，并不等于行内代码完全不能用。

一些足够轻量、足够局部的内容，行内脚本依然有存在空间，比如：

- 很小的页面示例或教学演示
- 只在当前页面用一次的极短初始化代码
- 需要紧贴当前文档结构展示的最小实验代码

但只要脚本开始变长、涉及复用、需要多人维护，或者需要在多个页面共享，行内代码就会迅速暴露出问题。它让 HTML 和行为逻辑粘得太紧，后续拆分时成本会越来越高。

## 6. 把脚本拆成多个外部文件会不会更慢？

这要看浏览器环境和资源组织方式。

早期网络协议下，请求太多小文件可能会让页面更慢，所以大家常常把很多逻辑打成一个大包。在支持 HTTP/2 及以上协议的环境中，这个问题缓和了不少，因为浏览器可以更高效地并行获取多个资源。

::: tip

HTTP/2 的多路复用允许在单个 TCP 连接上并行传输多个资源，大幅减少了请求排队。

SPDY 作为实验性协议已被 HTTP/2 完全取代，HTTP/3 则进一步用 QUIC 替换了底层的 TCP。

:::

这时，把应用拆成多个轻量脚本文件反而可能有优势：

- 公共部分更容易缓存
- 不同页面只下载自己需要的那部分脚本
- 组件级拆分更利于替换和复用

当然，这不是说文件越碎越好。拆分是否合适，仍然要看缓存命中率、请求开销和项目结构，而不是机械地追求“越多越模块化”。

## 7. 选择行内还是外部时，最实用的判断标准是什么？

可以用一个很朴素的判断标准：这段代码是不是“页面资产”，还是“独立脚本资产”。

- 如果代码只服务于眼前这份示例文档，而且规模很小，行内写法可以接受
- 如果它会被复用、要单独维护、会持续演进，外部文件通常是更稳的选择

你也可以把它简化成下面这张表：

| 情况               | 更合适的选择 |
| ------------------ | ------------ |
| 极短的单页示例     | 行内脚本     |
| 多页面共享逻辑     | 外部文件     |
| 需要浏览器缓存     | 外部文件     |
| 需要长期维护和重构 | 外部文件     |

默认把正式代码放进外部文件，行内脚本只保留给足够小、足够局部、足够短命的场景。
