## 3. 🤔 为什么表单需要 JavaScript 增强？

Web 表单很早就是 JavaScript 的重要应用场景。最初，JavaScript 经常用来在浏览器端做一部分验证，避免用户提交明显错误的数据，也减少服务器处理无效请求的压力。

即使今天有了更丰富的前端框架和 HTML5 表单能力，表单本身的核心模型仍然没有太大变化：用户在控件里输入数据，浏览器把字段名和值提交给服务器。JavaScript 的价值主要体现在这些地方：

- 在提交前验证数据，给出及时反馈。
- 改善控件的默认交互，例如自动聚焦、自动跳转、选择文本。
- 控制提交与重置，避免重复提交或误操作。
- 操作选择框、复选框、单选框等控件的状态。
- 将表单转换为适合 Ajax 提交的字符串或数据对象。
- 把富文本编辑区的 HTML 同步到真实表单字段中。

简单来说，HTML 负责提供表单结构，浏览器负责基本提交规则，JavaScript 负责把表单做得更好用、更可控。

## 5. 🤔 表单脚本有哪些容易踩坑的地方？

表单脚本的坑通常不是 API 难，而是行为时机和浏览器默认规则容易被忽略。

例如，用户点击提交按钮会触发 `submit` 事件，但直接调用 `form.submit()` 不会触发 `submit` 事件。因此，如果你把验证逻辑只放在 `submit` 事件里，手动调用 `submit()` 前就要自己先验证。

再例如，`change` 事件在不同控件上的触发时机不完全一样。文本框通常是在值改变并失去焦点后触发，选择框则通常在选择项改变时立即触发。不要把不同控件的 `change` 事件当成完全相同的东西。

还有一个常见问题是重复提交。用户如果点了多次提交按钮，服务器可能收到重复请求。浏览器端可以在表单提交后禁用提交按钮，但真正可靠的防重复提交还应该在服务端配合幂等设计。

## 6. 🤔 本章小结要抓住哪些结论？

Web 表单虽然很传统，但 JavaScript 仍然可以显著增强它的可用性。表单和表单字段暴露了属性、方法和事件，开发者可以用它们读取字段、设置焦点、监听变化、阻止提交、动态调整控件状态。

文本框是最常被增强的控件。你可以选择全部或部分文本，监听输入和剪贴板事件，限制输入格式，或者结合 HTML5 约束验证 API 让浏览器执行基础验证。

选择框也可以通过 JavaScript 灵活控制。DOM 方法让添加、移除、移动和重排 `<option>` 变得很直接，而选择框自己的 `options`、`selectedIndex` 和选项对象属性则适合读取和修改选择状态。

表单序列化的核心是模拟浏览器提交规则：只包含有名字且应该提交的字段，跳过禁用字段、普通按钮和未选中的单选或复选框，并对字段名和值做 URL 编码。

富文本编辑通常不是真正的表单字段。无论使用 `iframe` 的 `designMode`，还是使用 `contenteditable`，最终要提交给服务器时，都需要在提交前把编辑区的 HTML 同步到隐藏字段或其他可提交字段中。
