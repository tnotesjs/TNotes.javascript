## 3. 这一章到底在讲什么？

这一章讲的不是某个单独 API，而是一整套“顺序取值”的语言机制。

JavaScript 过去当然也能循环，但很多时候你必须先知道数据结构的内部访问方式，比如数组要靠索引、对象要靠键、集合要靠特定方法。ECMAScript 6 做的事情，是把这类顺序访问抽象成了统一协议。

从这个角度看，本章的核心问题其实只有一个：怎样让不同数据结构都能用一致的方式被遍历和消费。

## 4. 可迭代对象、迭代器和生成器之间是什么关系？

这三个概念最容易混在一起，但它们各自解决的问题不同。

| 概念       | 它回答的问题                   |
| ---------- | ------------------------------ |
| 可迭代对象 | “我能不能被顺序遍历？”         |
| 迭代器     | “下一项是什么？”               |
| 生成器     | “我能不能更方便地写出迭代器？” |

可迭代对象通过 `Symbol.iterator` 暴露默认迭代器工厂；迭代器通过 `next()` 一步步吐出值；生成器则是一种更适合编写暂停 / 恢复逻辑的函数结构，经常被拿来实现默认迭代器。

## 5. 为什么说这章会影响后面很多语法的理解？

因为很多你平时用得很顺手的语法，背后都依赖这套机制：

- `for...of`
- 数组解构
- 扩展操作符
- `Array.from()`
- `new Set(iterable)`
- `new Map(iterable)`
- 生成器里的 `yield*`

这些特性看起来分散，底层却在共享同一套“可迭代协议”。所以这一章不是孤立知识点，而是很多现代语法的共同底座。

## 6. 这一章应该怎样收束成稳定结论？

可以把这一章收束成下面几条：

- 迭代不是循环语法本身，而是一种按顺序消费数据的模式。
- 可迭代对象负责声明“我可以被遍历”。
- 迭代器负责按次序返回值，并通过 `done` / `value` 表达状态。
- 生成器让暂停和恢复执行变得简单，因此特别适合实现迭代器。
- `yield*` 可以把另一个可迭代对象直接接入当前生成流程。

把这几条连起来，你就能把 `for...of`、集合遍历和生成器写法放进同一张脑图里理解。
