## 3. 🤔 这一章到底在解决什么问题？

这一章其实在回答三个连续的问题：值是什么、代码在哪里找值、值什么时候会被回收。

先看值。JavaScript 变量里既可能放原始值，也可能放引用值。两者在复制、传参和类型判断时表现不一样。

再看查找。代码执行时并不是凭空就能拿到变量，而是依赖执行上下文和作用域链来逐层解析标识符。

最后看回收。变量和对象不是永远存在的，JavaScript 引擎会根据可达性和引用关系回收不再使用的内存。

所以，这一章不是三个互不相关的知识点，而是一条完整链路：变量绑定值，代码在作用域里访问值，引擎在合适的时候回收值。

## 4. 🤔 为什么要把变量、作用域和内存放在一起理解？

因为它们描述的是同一件事在不同阶段的表现。

- 变量阶段：你给一个名字绑定一个值。
- 作用域阶段：运行时决定这段代码能不能访问这个名字。
- 内存阶段：当这个值再也访问不到时，引擎决定何时释放它。

举个简单例子：

```js
function createUser(name) {
  const user = { name }
  return user
}

let currentUser = createUser('Ada')
currentUser = null
```

这里 `user` 是函数局部变量，只能在函数上下文里访问；返回后，外部通过 `currentUser` 持有这个对象；当 `currentUser` 被置为 `null` 后，这个对象就更容易在后续垃圾回收中被清理。

也就是说，变量决定“怎么命名”，作用域决定“谁能访问”，内存管理决定“什么时候消失”。

## 5. 🤔 这一章最核心的三条主线是什么？

可以把本章拆成下面三条主线：

| 主线         | 你要抓住的结论                             |
| ------------ | ------------------------------------------ |
| 值的层面     | 原始值和引用值在属性、复制、传参上行为不同 |
| 作用域层面   | 标识符解析依赖执行上下文和作用域链         |
| 生命周期层面 | 不再可达的值会在垃圾回收时被清理           |

围绕这三条主线，你至少要记住下面几个判断：

- 原始值不能像对象那样稳定地动态挂属性。
- ECMAScript 的函数参数一律按值传递。
- `typeof` 更适合判断原始类型，`instanceof` 更适合判断引用类型。
- `var` 是函数作用域，`let` 和 `const` 是块级作用域。
- 垃圾回收的核心不是“立刻释放”，而是“在合适时机回收不可达值”。

## 6. 🤔 学这一章时最容易误判什么？

最常见的误判有四类。

第一类是把“对象能被改”理解成“对象参数按引用传递”。这不对，参数仍然是按值传递，只不过这个值恰好是一个引用。

第二类是把“能访问变量”理解成“变量就存在于当前代码块里”。实际上，当前代码往往是在沿作用域链向上查找。

第三类是把“离开作用域”理解成“立刻回收内存”。垃圾回收不会在你写完这一行代码后立刻执行，它只是在后续某次回收时处理这些值。

第四类是把“删除属性”当成性能优化。很多时候，把引用设为 `null` 比频繁 `delete` 更稳妥，也更符合现代引擎的优化路径。

## 7. 🤔 这一章应该怎样收束成稳定结论？

可以把本章收束成下面几组结论。

关于值：

- JavaScript 变量可以保存原始值或引用值。
- 原始值复制的是内容副本，引用值复制的是指向同一对象的引用。
- `typeof` 适合看原始类型，`instanceof` 适合看对象与构造函数的关系。

关于作用域：

- 代码执行会进入全局、函数或块级作用域语境。
- 标识符解析总是从当前作用域开始，找不到再沿作用域链向上查。
- `var`、`let`、`const` 的差异，本质上是声明位置和生命周期管理的差异。

关于内存：

- JavaScript 依赖垃圾回收自动管理内存。
- 主流策略是标记清理，而不是早期引用计数。
- 全局引用、闭包、定时器和 DOM 关联对象，都是你需要格外小心的内存风险点。

把这三部分连起来，你对 JavaScript 运行时的理解就会完整很多。
