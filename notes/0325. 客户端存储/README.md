## 2. 评价

## 3. 为什么需要客户端存储？

Web 应用越来越像本地应用之后，页面经常需要保存和当前用户有关的数据。例如登录状态、主题偏好、临时表单草稿、离线数据、最近浏览记录和缓存的接口结果。

如果所有状态都依赖服务器，每一次页面刷新或交互都要重新请求，体验和性能都会受到影响。客户端存储提供了一条补充路径：把一部分数据留在用户设备上，让页面刷新、跳转甚至关闭后仍然能恢复某些状态。

最早广泛使用的方案是 `cookie`。后来，Web Storage 和 IndexedDB 提供了更明确的本地存储能力，让开发者不必把所有客户端数据都塞进会自动随请求发送的 `cookie` 里。

## 4. 三种客户端存储方案有什么区别？

| 方案 | 数据模型 | 生命周期 | 是否随请求发送 | 适合场景 |
| --- | --- | --- | --- | --- |
| `cookie` | 字符串名值对。 | 会话或指定过期时间。 | 会自动随匹配域和路径的 HTTP 请求发送。 | 会话标识、服务端需要读取的小型状态。 |
| `sessionStorage` | 字符串名值对。 | 当前页面会话。 | 不会自动发送。 | 单个标签页中的临时状态。 |
| `localStorage` | 字符串名值对。 | 持久保存，直到脚本或用户清除。 | 不会自动发送。 | 小型持久偏好和缓存。 |
| IndexedDB | 结构化对象存储。 | 持久保存，直到脚本或用户清除。 | 不会自动发送。 | 大量结构化数据、离线数据、可查询数据。 |

简单来说：服务端每次请求都需要的小型状态可以考虑 `cookie`；只在前端使用的小型字符串数据用 Web Storage；结构化数据、数据量较大或需要索引查询时，用 IndexedDB。

## 5. 客户端存储有哪些共同边界？

客户端存储通常和源绑定。协议、域名和端口共同决定一个源，不同源之间不能直接共享同一份 Web Storage 或 IndexedDB 数据。`cookie` 也有域和路径约束，浏览器会根据这些属性决定什么时候发送。

这些存储能力也都存在容量限制。`cookie` 很小，通常只适合保存少量字符串。Web Storage 常见限制是每个源数 MB。IndexedDB 的空间更大，但仍然受浏览器、设备和用户设置影响。

另外，客户端数据不是安全保险箱。用户可以清除，浏览器可能在存储压力下回收，脚本也可能在发生 XSS 后读取本地可访问的数据。敏感信息不应该明文保存在客户端。

## 6. 选择存储方案时应该看什么？

可以先问几个问题：

1. 服务器是否需要在每次请求中自动收到这份数据？如果需要，考虑 `cookie`。
2. 数据是否只在当前标签页有效？如果是，考虑 `sessionStorage`。
3. 数据是否只是少量字符串配置？如果是，考虑 `localStorage`。
4. 数据是否是大量对象、需要索引或离线查询？如果是，考虑 IndexedDB。
5. 数据是否敏感？如果敏感，优先放在服务端，客户端只保存必要标识，并配合 HTTPS、`HttpOnly`、`Secure`、`SameSite` 等安全策略。

选择存储方案不是只看容量大小，还要看数据是否会被自动发送、是否需要查询、是否跨页面共享、是否能承受同步阻塞、是否有安全风险。

## 7. 本章小结要抓住哪些结论？

Web Storage 定义了 `sessionStorage` 和 `localStorage`。前者用于保存一次页面会话期间的数据，后者用于跨会话持久保存数据。它们都只保存字符串，不会自动随 HTTP 请求发送。

IndexedDB 是浏览器中的结构化数据存储机制。它不是用表格保存数据，而是用对象存储保存 JavaScript 对象，并通过键、游标和索引来读取、遍历和查询数据。

这些能力让浏览器可以保存相当多的数据，但客户端数据默认不加密，也不天然可信。开发时要避免保存敏感信息，并且始终把关键权限和数据校验放在服务端。
