## 2. 评价

- 这一节看起来像概念铺垫，但它其实是在解决一个很实际的问题：你到底是在“写循环”，还是在“消费一个有顺序的数据结构”。这两件事看着像，抽象层级却不一样。

## 3. 迭代到底是什么意思？

迭代可以理解为：按照既定顺序，反复取出一组值并处理它们。

最简单的例子就是计数循环：

```js
for (let i = 1; i <= 10; ++i) {
  console.log(i)
}
```

它本质上就是一次次地产生值、消费值，只不过这里的值是数字序列。

当目标换成数组、集合或其他有顺序的数据结构时，迭代的核心仍然没变，只是“如何得到下一个值”这件事变复杂了。

## 4. 为什么手写索引循环并不理想？

对数组来说，索引循环当然能工作：

```js
const collection = ['foo', 'bar', 'baz']

for (let index = 0; index < collection.length; ++index) {
  console.log(collection[index])
}
```

但它有两个问题。

第一，你必须先知道这个数据结构是怎么访问的。数组靠索引，别的结构未必。

第二，遍历顺序是你手动硬编码出来的，不是数据结构自己声明出来的。

也就是说，这种写法把“数据本身的顺序能力”和“你如何遍历它”绑死在一起了。

## 5. `forEach()` 为什么还不够？

`forEach()` 确实把“手动维护索引”这件事简化了：

```js
const collection = ['foo', 'bar', 'baz']
collection.forEach((item) => console.log(item))
```

但它仍然有明显边界。

- 它主要还是数组方法，不是通用迭代协议。
- 它的回调形式比较固定。
- 它不适合表达更底层的“下一项怎么来”。

换句话说，`forEach()` 只是让数组遍历更方便了一点，并没有解决“不同数据结构如何统一被遍历”的问题。

## 6. 为什么 ES6 还要引入正式迭代机制？

因为语言需要一套统一标准，让不同数据结构都能被一致地消费。

只要某个对象声明自己是可迭代的，那么：

- `for...of` 可以遍历它
- 扩展操作符可以展开它
- 解构可以消费它
- `Array.from()` 可以从它创建数组

这套机制让“如何取值”变成数据结构和协议本身的能力，而不是每次都让使用者手写一套专用循环。

所以这一节最重要的结论是：迭代真正要解决的，不是“怎么写循环”，而是“怎样统一顺序取值的接口”。
