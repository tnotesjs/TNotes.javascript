## 3. 🤔 Ajax 解决了什么问题？

Ajax 的核心目标，是让页面不刷新也能和服务器交换数据。用户点击按钮、切换列表、提交局部表单时，页面可以只请求需要的数据，再用 DOM 更新局部内容。

Ajax 这个名字来自 Asynchronous JavaScript + XML，但实际使用并不依赖 XML。它真正重要的不是数据格式，而是“异步请求 + 局部更新”的交互模型。JSON、纯文本、HTML 片段、Blob、表单数据都可以作为通信载荷。

XHR 是早期 Ajax 的核心对象。它给浏览器提供了原生请求能力，让开发者不用再依赖隐藏窗格、内嵌窗格、Flash 或其他绕路方案。后来 Fetch API 出现，用 Promise、`Request`、`Response`、`Headers` 和流式主体提供了更现代的网络接口。

## 4. 🤔 本章这些 API 分别适合什么场景？

| 能力 | 适合场景 | 重点 |
| --- | --- | --- |
| `XMLHttpRequest` | 旧项目、兼容历史浏览器、需要传统 XHR 事件模型。 | `readyState`、状态码、请求头、`FormData`、超时和进度事件。 |
| Fetch API | 现代 HTTP 请求、JSON 接口、文件上传、流式响应。 | Promise、`Response.ok`、`Headers`、`Request`、Body 读取和 `AbortController`。 |
| CORS | 浏览器正式跨源访问。 | 服务端通过 HTTP 头部授权浏览器读取跨源响应。 |
| 图片探测和 JSONP | 遗留跨源通信。 | 不需要 CORS，但能力有限且安全风险更高。 |
| Beacon API | 页面隐藏或卸载时发送遥测数据。 | 不阻塞页面导航，不读取响应。 |
| WebSocket | 低延迟、全双工、长连接通信。 | 聊天、协作、实时状态、行情和游戏同步。 |

实际开发中，普通请求优先考虑 `fetch()`，需要实时双向通信再考虑 WebSocket，需要退出页面时可靠上报再考虑 `sendBeacon()`。

## 5. 🤔 同源策略为什么贯穿本章？

浏览器网络请求不只是技术问题，也是安全问题。默认情况下，脚本发起的请求受同源策略限制。协议、域名和端口都相同，才算同源。

同源策略限制了脚本随意读取其他站点响应的能力。否则，恶意页面就可能利用用户浏览器中已有的登录态，读取用户在另一个网站上的隐私数据。

CORS 是浏览器提供的正式跨源机制。它不是绕过安全限制，而是让服务器通过响应头明确告诉浏览器：哪些来源可以读取响应、哪些方法和头部被允许、是否允许携带凭据。

## 6. 🤔 本章小结要抓住哪些结论？

Ajax 让 Web 页面可以在不刷新的情况下从服务器获取数据。XHR 是 Ajax 普及的中心对象，虽然 API 较老，但基本模型影响了很久。

XHR 的主要限制来自同源策略。CORS 是正式跨源方案，图片探测和 JSONP 是更早的替代方案，但它们能力有限，也不如 CORS 可靠。

Fetch API 是 XHR 的现代替代方案，接口更清晰，基于 Promise，支持 `Request`、`Response`、`Headers` 和 Streams API。Beacon API 解决页面即将离开时的轻量上报问题。WebSocket 则提供了不基于 HTTP 请求响应循环的双向长连接。

最后，所有网络请求都必须考虑安全。前端能请求某个 URL，不代表用户有权访问那里的数据。真正的权限校验、CSRF 防护和数据保护都必须在服务端落实。
